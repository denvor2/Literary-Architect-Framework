## Контекст

ADR-0012 Decision 5 (Accepted, отражает явное решение Product Owner 2026-07-10): "пользователь
должен быть явно предупреждён" при недоступности БД/рассинхроне — это была прямая коррекция
Product Owner первоначальной рекомендации "тихий fallback", записанная в ADR как обязательное
требование, не опция. Sprint-24-Step-06's Step Card (написана sprint-planner до того, как это
решение было внесено в ADR-0012) ошибочно исключила любые визуальные индикаторы из Definition of
Done — из-за этого несоответствия видимого предупреждения в реализации Sprint 24 до этого шага
не было вообще. Этот шаг закрывает разрыв между тем, что ADR-0012 требует, и тем, что было
реализовано.

Зависит от Sprint-24-Step-07 (docs/task-bus/queue/pending или active/done в зависимости от
момента исполнения) — использует его `getSyncWarning()` (или как он будет назван по факту
реализации Step 07 — прочитай реальный код, не это описание).

## Objective

1. `useWorkspaceController.ts`: добавить новое поле в возвращаемый хуком объект (Sprint-24-
   Step-06 сознательно это запретил — тот запрет снят явным решением Product Owner 2026-07-11,
   не регресс, а осознанное расширение). Поле отражает текущее предупреждение
   (`getSyncWarning()` из Step 07) — обновляется после каждого `loadWorkspace()`/
   `saveWorkspace()` вызова в уже существующих useEffect'ах.

2. Минимальный видимый UI-элемент (без новой библиотеки toast/notification — в проекте такой
   нет, не добавлять новую зависимость ради этого): простой баннер/полоска в верхней части
   приложения, видимая когда предупреждение активно, с понятным текстом на русском:
   - "db-unavailable" (или как оно будет названо) -> например "База данных временно
     недоступна — изменения сохраняются локально."
   - "recovered-local-wins" (одноразовое) -> например "Обнаружено расхождение с базой данных
     — использованы более свежие локальные изменения." Исчезает после следующего успешного
     сохранения или через разумный таймаут (на усмотрение реализации — не проектировать
     сложную систему уведомлений, один текстовый баннер достаточно для Sprint 24).

3. Не имитировать полноценный design system компонент — цвет/стиль по образцу уже существующих
   в проекте предупреждений/ошибок (посмотри, например, как показываются ошибки Line
   Editor/Critic в AssistantPanel.tsx, если там есть подходящий паттерн, для консистентности).

## Rules

- Не добавлять никакой новой библиотеки (toast/notification) — простой conditional-render
  баннер на чистом Tailwind, как остальной проект.
- Не проектировать вперёд систему уведомлений с очередью/историей — один текущий статус,
  простое условие показа/скрытия.
- Не менять форму данных, которые видит остальной код через хук, кроме добавления этого
  одного нового поля.

## Validation

- npx tsc --noEmit, npm run lint, npx prettier --check, npm run build — чисто.
- Живая проверка (та же изолированная техника — отдельный scratch-порт/тестовая БД, НЕ
  localhost:3000 напрямую с реальными данными Product Owner, если только сам Product Owner не
  подтвердит, что его вкладка сейчас свободна для теста):
  1. Остановить Postgres, отредактировать сцену -> баннер "БД недоступна" появляется.
  2. Восстановить Postgres, перезагрузить страницу -> баннер меняется на "расхождение
     устранено"/скрывается по логике Step 07, локальные правки видны (не потеряны — совместно
     с проверкой Step 07).
  3. Обычная работа без сбоев БД -> баннер не показывается никогда (не ложное срабатывание).
- Скриншот или явный текстовый вывод (например, innerText найденного баннера через Playwright)
  в ARP как подтверждение, что баннер реально виден в DOM, а не просто "код должен его
  показать".
