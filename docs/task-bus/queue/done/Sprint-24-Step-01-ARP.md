## Что сделано

Написан и принят `docs/adr/ADR-0012-persistence-migration.md` — по формату/тону `ADR-0011`
(Status/Date/Deciders/Relates to → Context → Decision → Consequences → Known Gaps → Review
Trigger). ADR явно ссылается на `ADR-0003` (Persistence: PostgreSQL/Prisma — Phase 2+) и на
заголовочный комментарий `prisma/schema.prisma` ("Workspace UI state ... is NOT persisted here").

ADR закрывает все шесть вопросов Step Card отдельными подразделами Decision:

1. **Модель пользователя** — единственный локальный пользователь на инстанс (find-or-create,
   без логина), явно помечено как временная заглушка со сроком замены не позже Sprint 29
   (мультипользовательская система, ссылка на запись Sprint 28 в ROADMAP_18-27.md).
2. **Разделение персистентности** — в БД уходит только `books`; `localStorage` продолжает хранить
   весь `Workspace` целиком, включая эфемерные поля, без изменений.
3. **Форма API** — единый "грубый" эндпоинт `/api/workspace` (GET/PUT всего дерева), а не
   гранулярный REST; repository-слой может быть декомпозирован по сущностям внутри — это деталь
   реализации, не влияющая на внешний контракт.
4. **Идентификаторы сущностей** — зафиксирована находка (локальные `String(nextNumber)` id
   коллидируют между разными книгами/главами при глобально уникальном `@id` Prisma) как
   предпосылка; само исправление явно вынесено в `Sprint-24-Step-02`, выполняемый до
   repository-слоя (`Sprint-24-Step-03`).
5. **Dual-mode: обнаружение недоступности БД и рассинхрон** — проверка на каждый вызов (не
   только на старте сессии); неудачная запись → тихий fallback на `localStorage` на уровне
   хранилища, НО обязательное видимое предупреждение пользователю (конкретная форма — на
   усмотрение Step 06); рассинхрон — last-write-wins; здоровье БД/бэкап зафиксированы как
   требование к будущему Admin-функционалу, не реализуются в Sprint 24.
6. **Механизм переноса данных** — единственный технически осуществимый вариант: автоматический
   перенос в браузере при загрузке приложения (пустая БД + непустой `localStorage` → `PUT
   /api/workspace`), не отдельный CLI-скрипт.

**Важно (явно отмечаю по требованию Output Step Card):** решения по пунктам 1 и 5 — это были
продуктовые развилки, уже подтверждённые Product Owner 2026-07-10 ДО начала этого шага (текст
решений дословно взят из Step Card). Я не вырабатывал эти решения самостоятельно — только
перенёс их в ADR-формат (Context/Decision/Consequences/Review Trigger) и связал с остальными
четырьмя техническими констатациями (пункты 2, 3, 4, 6), которые уже были однозначно определены
по коду/существующим документам и не требовали продуктового решения.

ADR получил статус **Accepted** сразу (не "Proposed") — согласно явному указанию Step Card
("дополнительного раунда согласования по ним не требуется").
