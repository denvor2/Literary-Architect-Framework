## Контекст

Product Owner попросил (устно, 2026-07-11) добавить форму поиска в верхнюю строку меню
(apps/studio/src/components/Header.tsx, добавлена Sprint-25-Step-01 как chrome-only заглушка —
Файл/Правка/Вид disabled, переключатель языка и "Войти" disabled) — поиск "по текстам книги
и книг в целом": (a) по главам/сценам текущей открытой книги, (b) по списку книг рабочего
пространства (найти/переключиться на книгу по названию). Product Owner явно попросил именно
одну форму ("форма для поиска", единственное число) — не два отдельных поля.

Это первый функциональный элемент в Header.tsx — сегодня компонент не принимает пропсов вообще
(export function Header()) и не имеет доступа ни к Workspace, ни к книгам. Вся необходимая
информация уже целиком в памяти клиента (useWorkspaceController.ts, books: Book[], activeBookId,
chapters = activeBook.chapters, каждая Chapter содержит scenes: Scene[] с text,
domain/model.ts) — новый API-роут/запрос к БД не нужен, поиск — чистый клиентский
substring-фильтр по уже загруженным данным.

Небольшая, полностью клиентская задача без изменений домена/схемы/бэкенда — поэтому один Step
Card, не последовательность "плумбинг -> UI" (в отличие от, например, Sprint 24, где через API
и БД реально проходили новые данные). Разбивать её на несколько карточек было бы искусственным
дроблением одного связного среза.

## Разобранные технические развилки (решены самостоятельно, не эскалируются)

1. Форма UI: один input, общий выпадающий список с двумя подписанными секциями ("Книги" и
   "Главы и сцены"), а не два раздельных поля и не отдельные экраны поиска. Это сохраняет ровно
   один видимый поисковый элемент (буквальное требование PO), при этом результаты остаются
   различимыми по типу — паттерн, стандартный для единого quick-open поиска (аналогично тому,
   как Sprint-25-Step-01 уже вводит выпадающие меню Файл/Правка/Вид с тем же
   click-outside-to-close механизмом, который здесь переиспользуется).
2. Логика поиска — чистая функция в apps/studio/src/domain/search.ts (новый файл), не размазана
   по Header.tsx. domain/model.ts сегодня документирует себя как "pure types only, no UI
   logic" — чистая, не мутирующая функция сопоставления над Book/Chapter/Scene не нарушает этот
   принцип (это не UI-логика и не побочный эффект), и даёт то же преимущество, что и остальной
   домен: тестируется Node-скриптом с фикстурными данными без браузера (см. Validation, Shape 2
   из literary-studio-live-verify).
3. Что именно ищем (v1): Book.title — по всем книгам рабочего пространства (переключение);
   Chapter.title, Chapter.subtitle, Scene.title, Scene.text — только у активной книги (PO
   описал это как "текст книги", то есть содержимое текущей открытой книги, не всех книг сразу).
   Простое регистронезависимое вхождение подстроки (toLowerCase + includes) — никакой
   fuzzy-библиотеки и поискового индекса не добавляется (простого substring-фильтра достаточно
   для объёма данных, который целиком уже в памяти клиента). Известное ограничение, не решается
   в этом шаге: не учитывает "ё"/"е" и другие орфографические варианты — обычный
   String.includes, ничего специального под кириллицу.
4. Персонажи/Идеи — НЕ ищутся в v1 (см. "Открытый вопрос" ниже — не блокирует этот шаг,
   зафиксировано как явное решение, не молчаливое умолчание).
5. Переход к найденному месту переиспользует существующий механизм скролла Sprint 16-17
   (функция scrollBlockIntoView в Sidebar.tsx, id вида chapter-block-{id}/scene-block-{id},
   EditorArea.tsx (UnifiedBookView) — эти id уже проставлены на каждом блоке главы/сцены). Не
   изобретается новый способ навигации. Реализация: та же 5-строчная функция scrollBlockIntoView
   не импортируется из Sidebar.tsx (не трогаем этот файл вообще, см. Forbidden paths) — копия
   той же функции заводится в page.tsx, где и так уже нужна новая функция-обработчик (см. пункт
   6). Дублирование пяти строк дешевле, чем выделение общего модуля ради одного второго
   вызывающего места, и не расширяет Scope на стабильный, не относящийся к этой задаче
   Sidebar.tsx.
6. Переключение книги переиспользует существующий selectBook/handleSelectBook (уже есть в
   page.tsx, передаётся в Sidebar) — новый механизм переключения книг не создаётся. Клик по
   собственной активной книге в результатах ведёт себя так же, как клик по ней в списке книг
   сайдбара (deselectAll — "вернуться к обзору книги"), это не новое решение.
7. Сворачивание/раскрытие перед скроллом. Главы/сцены на едином виде книги (UnifiedBookView)
   могут быть свёрнуты (collapsedChapterIds/collapsedSceneIds/isChaptersCollapsed,
   Sprint 16-17-Step-03, состояние поднято в page.tsx) — если совпадение находится внутри
   свёрнутого блока, клик по результату поиска обязан сначала развернуть нужную главу/сцену (и
   сбросить общее "свернуть все главы", если оно активно), иначе scrollIntoView либо промахнётся
   мимо всё ещё скрытого текста, либо (при полностью свёрнутой главе) не найдёт целевой
   scene-block-{id} в DOM вовсе. Новая функция handleSelectSearchMatch(chapterId, sceneId?) в
   page.tsx обязана явно раскрыть нужные уровни через уже существующие сеттеры
   (setIsChaptersCollapsed/collapsedChapterIds/collapsedSceneIds) перед вызовом
   selectChapter/selectScene и скроллом.
8. Гонка рендера при переключении с CharacterPanel или с другой книги. Если в момент клика по
   результату поиска отображается CharacterPanel (selectedCharacterId установлен) или активна
   другая книга, selectChapter/selectScene уже сбрасывают selectedCharacterId (см.
   useWorkspaceController.ts), но целевой DOM-узел (chapter-block-{id}/scene-block-{id})
   физически не существует в момент вызова обработчика — он появится только после следующего
   рендера EditorArea. Функция scrollBlockIntoView обязана вызываться не синхронно сразу после
   стейт-сеттеров, а через requestAnimationFrame (двойной requestAnimationFrame, если один
   недостаточно надёжен на практике — проверить живьём, см. Validation) — иначе клик по
   результату поиска, ведущему в другую книгу или от CharacterPanel, будет молча не долистывать
   до цели. Это тот же самый класс гонки, что уже потенциально скрыт в сегодняшнем обработчике
   Sidebar.tsx (который вызывает скролл синхронно) — но чинить Sidebar.tsx отдельно вне scope
   этого шага (Forbidden paths); здесь важно не унаследовать ту же проблему в новом, значительно
   более часто задействующем этот переход коде.
9. Минимальная длина запроса перед показом результатов — 2 символа (обрезанные пробелы не
   считаются), чтобы не выводить весь список книг/глав на первую нажатую букву. Пустой ввод —
   выпадающий список закрыт.
10. Сниппет текста сцены в выпадающем списке — простая вырезка примерно 40 символов вокруг
    первого вхождения (без библиотек подсветки), чтобы результат был не просто "Сцена 3" без
    контекста совпадения. Часть domain/search.ts, не отдельная функциональность.

## Rules

- Простое регистронезависимое вхождение подстроки (toLowerCase + includes) — никакой
  fuzzy-библиотеки, никакого поискового индекса, никакого нового npm-пакета.
- Минимум 2 непробельных символа в запросе перед показом результатов; пустой/короткий запрос —
  выпадающий список закрыт, ничего не подсвечено в шапке лишним UI-шумом.
- Выпадающий список — две подписанные секции ("Книги" и "Главы и сцены"), в этом порядке или
  обратном (на усмотрение реализации, не принципиально) — не смешивать без подписи, чтобы было
  понятно, что произойдёт по клику (переключить книгу или проскроллить к месту).
- Клик по результату закрывает выпадающий список (после навигации) и не оставляет фокус в поле
  ввода в состоянии, будто поиск всё ещё активен — но само значение запроса можно не сбрасывать
  (не принципиально, при следующем открытии выпадающего списка запрос уже будет виден).
- Escape и клик вне формы поиска закрывают выпадающий список — тот же паттерн, что уже есть в
  Header.tsx для Файл/Правка/Вид (mousedown-обработчик на внешний клик, menuBarRef) — расширить
  существующий useEffect/ref, либо завести отдельный, если проще для локального состояния
  поиска: не изобретать новый способ закрытия по клику вне, если уже есть подходящий рядом.
- handleSelectSearchMatch в page.tsx обязан: (1) снять "свернуть все главы", если оно активно;
  (2) снять свёрнутость нужной главы, если она свёрнута; (3) для совпадения сцены — также снять
  свёрнутость нужной сцены; (4) вызвать selectChapter/selectScene; (5) выполнить скролл через
  requestAnimationFrame (не синхронно) — см. развилки 7-8 выше.
- Переключение книги через результат поиска — прямой вызов уже существующего в page.tsx
  handleSelectBook(bookId), без нового кода переключения.
- Клавиатурный шорткат для фокуса поля поиска (например, Ctrl+K или /) — необязательное
  улучшение, включить только если реализация тривиальна (глобальный keydown-listener,
  игнорирующий события, когда фокус уже в другом input/textarea/contenteditable) — не блокирует
  Definition of Done этого шага, если окажется, что требует больше работы/тестирования, чем
  ожидалось (конфликты с чужими шорткатами и т.п.) — пропустить и явно отметить в ARP, что
  пропущено и почему.
- UI-текст — по-русски, тем же визуальным языком, что остальной Header.tsx (zinc/rounded-md/
  border вокабуляр, тот же паттерн выпадающего блока, что у Файл/Правка/Вид).
- Не переделывать структуру шапки (лого/меню/языковой переключатель/"Войти") — форма поиска
  добавляется как ещё один элемент строки, не заменяет и не двигает уже существующие.

  изолированную песочницу без риска для реальных данных Product Owner. Через Playwright

  1. Создать тестовую книгу с 2+ главами, 2+ сценами с разным русскоязычным текстом, и вторую
     книгу с отличающимся названием.
  2. Ввести в поиск подстроку, встречающуюся только в тексте одной сцены — убедиться, что в
     выпадающем списке появляется именно она (секция "Главы и сцены"), клик по ней скроллит к

     page.locator(...).isVisible() и позицию скролла, не просто что клик не упал с ошибкой).
  3. Свернуть главу, содержащую целевую сцену (или включить "свернуть все главы"), затем
     повторить клик по тому же результату поиска — убедиться, что глава/сцена автоматически
     разворачивается и скролл всё равно долистывает до цели (см. развилка 7).
  4. Открыть CharacterPanel (кликнуть персонажа в сайдбаре, если создан, либо создать одного),
     затем кликнуть результат поиска по сцене — убедиться, что происходит переход обратно к
     EditorArea с корректным скроллом (см. развилка 8 — гонка рендера).
  5. Ввести подстроку, встречающуюся только в названии второй книги — убедиться, что она

    занятом порту.
