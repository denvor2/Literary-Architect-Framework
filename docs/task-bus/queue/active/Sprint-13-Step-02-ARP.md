# ARP: Sprint-13-Step-02

**Step Card:** Backend: все четыре route.ts принимают историю сообщений (messages)
**Тип:** implementation (backend, 4 файла)
**Исполнитель:** Programmer (Executor)

## Что сделано

Единая схема запроса применена во всех четырёх `route.ts`, сервер остаётся полностью
stateless (ADR-0004 без изменений) — вся история диалога приходит от клиента в `messages` при
каждом вызове:

### `/api/coauthor`

- `currentText` переименован в `sceneText` (единая схема со всеми четырьмя), `?? ""` как и
  предписывал Step Card дословно.
- `bookContext` остался обязательным.
- Валидация `messages`: массив, каждый элемент — `{ role: "user" | "assistant", content: string
  }`, иначе 400 с конкретным сообщением.
- Собирается `contextMessage` (книга + текущий текст сцены, как раньше был единственным
  сообщением) + `...messages` поверх него.
- System-промпт дополнен: явно указано, что это продолжающийся диалог, а не разовый запрос;
  если `messages` пуст — переходить прямо к черновику/продолжению сцены.

### `/api/line-editor`

- `text` → `sceneText`, остался обязательным (Editor всегда работает с конкретным текстом).
- `bookContext` остался опциональным (Sprint 12, без изменений).
- Та же валидация `messages`.
- `contextMessage` — как раньше (с `bookContext` или без), `...messages` добавлены поверх.
- System-промпт дополнен: если автор задаёт уточняющий вопрос о правке — отвечать на него
  напрямую, иначе — только отредактированный текст, как и было.

### `/api/critic`

- `text` → `sceneText`, обязателен. **`bookContext` не добавлен вообще** (ни опционально) — по
  прямому указанию Step Card, Critic/Reader вне таблицы контекстов ADR-0008.
- Та же валидация `messages`.
- `contextMessage` — просто `sceneText` без обёртки (как и раньше).
- System-промпт дополнен: на уточняющие вопросы про уже данный отзыв — отвечать записью в той
  же структуре (`category: "General"`, `severity` по значимости ответа, `comment` — сам ответ),
  чтобы форма ответа (`{ ok, reviews }`) не менялась ни при каком сценарии диалога. Это
  самостоятельное дизайн-решение с моей стороны — Step Card не расписывал точный механизм
  сохранения структурированной формы при диалоге, я выбрал минимальный вариант, укладывающийся
  в уже существующую схему `reviews`.

### `/api/reader`

- `text` → `sceneText`, обязателен. Без `bookContext`, как и Critic.
- Та же валидация `messages`, тот же паттерн `contextMessage` + `...messages`.
- System-промпт дополнен: на уточняющие вопросы о своей реакции — отвечать напрямую, оставаясь
  тем же читателем, а не переключаясь в редактора/критика.


**Живая проверка — все четыре route.ts, все три обязательных сценария, реальный (не
замоканный) `next dev` сервер, реальные ответы Anthropic** (использовался уже запущенный в этом
окружении сервер на порту 3000; отдельная попытка поднять ещё один на 3131 корректно
отклонилась Next.js как дубликат — не мешало проверке).

Технический нюанс: Cyrillic-текст, переданный прямо аргументом `curl -d '...'` в этой
Windows/Git-Bash среде, ломался кодировкой на уровне командной строки (не в самом route.ts) —
решилось записью JSON-тела в файл и `curl --data-binary @file`, тот же обходной приём, что уже
использовался в предыдущих сессиях для Cyrillic-контента.

### `/api/coauthor`

- **(a) `messages: []`** — пустой `sceneText`, `bookContext` с реалистичной книгой → получен
  осмысленный черновик новой сцены (не игнорирует контекст: использует имя героини).
- **(b) `messages` с 3 репликами** (найти амулет → ассистент описал находку → «пусть заметит
  царапины») → ответ прямо продолжил с того места, учёл предыдущую реплику ассистента
  (амулет, полумесяц) и добавил именно запрошенную деталь (царапины, привязка к архивным
  документам) — история не игнорируется.
- **(c) невалидные формы:** `messages: "not-an-array"` → 400 `"messages must be an array."`;
  `messages: [{"role":"weird",...}]` → 400 про role/content; отсутствие `bookContext` → 400
  (сохранённое поведение Sprint 12).

### `/api/line-editor`

- **(a)** `messages: []`, `sceneText` с двумя намеренными ошибками согласования → обе
  исправлены («сел»→«села», «красный»→«красные»), больше ничего не тронуто.
- **(b)** диалог из 3 реплик (вопрос «почему исправил» → ответ ассистента → «проверь ещё раз на
  другие ошибки») → модель вернула тот же корректный текст без новых правок — учла, что правка
  уже сделана и подтверждена в истории, а не начала править заново с нуля.
- **(c)** `messages` не массив → 400; отсутствие `sceneText` → 400 `"No sceneText provided."`.

### `/api/critic`

- **(a)** `messages: []` на намеренно слабом тексте (телеграфный стиль, штампы) → структурированный
  `reviews` с конкретными замечаниями по категориям Style/Pacing/Characters/Plot/General.
  **Честно фиксирую:** первая попытка вернула `{"ok":false,"error":"Critic response was not
  valid JSON."}` — модель не выдержала формат "только сырой JSON" с первого раза; повтор того
  же запроса сразу дал валidный JSON. Это уже существовавшая характеристика Critic (не введена
  этим шагом) — тот же паттерн non-determinism, что и раньше, обрабатываемый существующим
  `try/catch` вокруг `JSON.parse`, а не новый регресс.
- **(b)** диалог из 3 реплик (вопрос про severity → ответ ассистента с JSON-фрагментом →
  «что исправить в первую очередь») → ответ учёл предыдущий отзыв, вернулся одной записью
  `category: "General"` с конкретной рекомендацией — подтверждает design-решение выше.
- **(c)** элемент `messages` без `content` → 400 про role/content; отсутствие `sceneText` → 400.

### `/api/reader`

- **(a)** `messages: []` на сцене с ножом → развёрнутая читательская реакция прозой, без
  списков/категорий, с вопросами "что дальше" — форма не изменилась.
- **(b)** диалог из 3 реплик («что зацепило» → ответ → «убежать или остаться») → ответ прямо
  продолжил разговор, ссылаясь на предыдущую реплику про крик, остался в роли читателя, не
  соскользнул в редактора/критика.
- **(c)** `content` не строка (число) → 400 про role/content; отсутствие `sceneText` → 400.

`git status --short` → только 4 изменённых `route.ts` (плюс сам Step Card, перемещённый в
`active/`) — `apps/studio/src/ai/**`, весь UI, `domain/**`, `storage/**` не тронуты.

## Отклонения от Step Card

Одно самостоятельное дизайн-решение (не отклонение от Scope/Forbidden, а заполнение
недосказанности): как Critic сохраняет структурированную форму ответа при диалоговом
уточняющем вопросе — решено единой записью `category: "General"` внутри уже существующей схемы
`reviews`, без изменения формы ответа. Описано выше в разделе `/api/critic`.
