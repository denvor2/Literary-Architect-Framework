## Роль и метод

Независимая перепроверка функциональных claim'ов ARP `Sprint-25-Step-01-ARP.md`, роль
`tester` (второй gate наравне с `architect-reviewer`, не пересекается с ним по предмету —
здесь только функциональная перепроверка, не архитектура/scope). Ничего из ARP не
переиспользовано: собственный `Playwright`-скрипт, собственный прогон статической валидации,
собственный сервер.

## Часть 0 — обязательная проверка утечки тестовых данных в реальной БД (сделано первым)

ARP прямо признаёт, что его собственный проверочный скрипт во время диагностики случайно
записал `"Тестовая книга light"`/`"Тестовая книга dark"` в реальный локальный Postgres (dual-mode
storage, ADR-0012 — scratch-сервер и обычный `npm run dev` используют один `DATABASE_URL`), и
заявляет, что это устранено и проверено через `GET`/`PUT /api/workspace`.

Проверено независимо, **до** какой-либо собственной активности с БД: `GET /api/workspace`
против уже работающего живого `npm run dev` Product Owner'а (порт 3000, только чтение) —

```
books: ["0. Wild Mind", "1 Слон, которого не было", "2 Законы Тералии", "3 Проект Уллис"]
```

Ни `"Тестовая книга light"`, ни `"Тестовая книга dark"`, ни любого другого явно тестового
названия — нет. **Утечки тестовых данных, о которой пишет ARP, на момент проверки в реальной
БД не обнаружено.**

Отдельно: список книг не совпадает по названиям с тем, что ARP называет "исходными" (`"Книга
1"`/`"Вторая книга 2"`) — это не признак порчи данных, а ожидаемое следствие того, что между
завершением работы ARP и этой проверкой Product Owner продолжал реально работать в живом
`npm run dev` (переименовывал книги, добавлял новые). Подтверждено это же в конце проверки (см.
раздел "Постусловия" ниже) — единственное расхождение между снимком БД до и после всей моей
работы затронуло книгу, которую я не трогал вообще, и совпадает по характеру с обычным
редактированием текста автором.

## Важная находка среды (не регрессия этого шага, но существенно для доверия к методике)

Стандартный приём `literary-studio-run`/ARP — `npm run build && npx next start -p <порт>` —
оказался **ненадёжен именно в этой среде**, когда `apps/studio/.next` одновременно
используется живым `npm run dev` Product Owner'а (порт 3000, работал всё время параллельно).
Наблюдалось воспроизводимо: `next start` (несколько независимых попыток, каждый раз новая
сборка) периодически отдавал `500` на случайный JS/CSS/шрифт-чанк, из-за чего клиентская
гидратация ломалась частично или полностью, и Sidebar молча показывал "Пока нет книг" при
том, что в реальной БД книги были — без какого-либо видимого предупреждения пользователю.
Не связано с диффом этого шага (файлы диффа не трогают `next.config.ts`; сам `next start`
явно печатает предупреждение `"next start" does not work with "output: standalone"
configuration`).

Устранено переходом на рекомендованный самим Next.js способ — `node
.next/standalone/server.js` — но и это стабильно заработало только после копирования
собранного `.next/standalone` + `.next/static` + `public` в **полностью отдельную от
`apps/studio/.next` директорию** (scratchpad), вне досягаемости живого dev-сервера. После
изоляции — 3 подряд чистых прогона без единого `requestfailed`.

Это стоит вынести Architect'у/Product Owner'у отдельно от вердикта по этой карточке: пока
`npm run dev` Product Owner'а работает параллельно, `npm run build && npx next start` в
`apps/studio/` — ненадёжный способ живой проверки, и ARP, скорее всего, просто не попал на эту
гонку в моменте своей проверки (либо попал и не заметил, так как транзиентная порча чанка не
всегда рвёт критичный путь).

## Часть 2 — независимое живое воспроизведение (light + dark, изолированный сервер)

Собственный Node/Playwright-скрипт (`@playwright/test`-совместимый API, `playwright-core`,
системный Chrome, `channel: "chrome"`), написан с нуля, не переиспользует скрипт ARP. Против
изолированного standalone-сервера (см. выше). Выбрана уже существующая книга `"0. Wild Mind"`
(id `"1"`) через реальный клик в Sidebar — новая книга не создавалась.

1. **Заголовок "Помощники"** — `textContent` (не `innerText`, чтобы не зависеть от
   CSS `uppercase`-transform) заголовка в `AssistantPanel` — точное совпадение `"Помощники"` в
   обоих режимах. Отдельно: `/\bAssistants\b/` не находит совпадений в `body.innerText` —
   английского текста нигде на странице нет.
2. **Секция "Идеи" в Sidebar** — присутствует в `<aside>`, идентифицированном строго по наличию
   заголовка "Персонажи" (чтобы не спутать с `<aside>` от `AssistantPanel`, у которого тоже есть
   свой `<textarea>` для чата — это отдельно обнаружено чтением `AssistantPanel.tsx` перед
   написанием скрипта и стало источником первой версии бага в самом скрипте, см. ниже). В
   `<main>`/`EditorArea` секция "Идеи и заметки" подтверждённо отсутствует.
3. **Полный CRUD-цикл идей на реальной книге**:
   - Клик "+ Добавить заметку" — заметка с пустым текстом создаётся немедленно (проверено
     чтением `createIdea()` в `useWorkspaceController.ts`: `{ text: "" }` пушится в массив сразу,
     без ожидания ввода) — placeholder "Пока нет заметок" корректно пропадает уже на этом шаге
     (edge case, которого не было в скрипте ARP).
   - Ввод текста в `<textarea>` — значение реально сохраняется в поле.
   - Реальное сохранение на сервер подтверждено **независимым** `GET /api/workspace`
     (опрошено с retry до 3с) — введённый текст физически присутствует в `ideas` нужной книги в
     БД, а не только в локальном React-состоянии.
   - Удаление — клик "Удалить", реально принят `window.confirm` (`page.once("dialog", accept)`),
     заметка пропадает из DOM.
   - Удаление подтверждено persisted: повторный независимый `GET /api/workspace` — текста
     заметки в БД больше нет.
4. **Строка меню Header**:
   - `innerText` шапки содержит `Файл`/`Правка`/`Вид`/`RU`/`Войти` в обоих режимах.
   - **Toggle одной кнопкой** (не проверено в скрипте ARP): повторный клик по "Файл" открывает,
     затем закрывает то же меню — подтверждено в обоих режимах.
   - **Переключение между двумя меню без промежуточного клика вне** (не проверено в скрипте
     ARP): клик "Файл" → сразу клик "Правка" — открытым остаётся ровно одно выпадающее меню, не
     два — подтверждено.
   - Пункт "Скоро" — реально `disabled` (DOM-атрибут, не только визуально приглушён) в каждом
     из трёх меню; принудительный клик (`force: true`) не роняет страницу и не производит
     побочных эффектов.
   - `RU` и `Войти` — реально `disabled` в обоих режимах.
   - **Найдено и зафиксировано как замечание, не как FAIL** (не проверено в скрипте ARP): клик
     вне `<nav>`, но внутри `<header>` (например, по надписи "Literary Studio") **не закрывает**
     открытое меню — обработчик "клик вне закрывает" повешен на `menuBarRef` (только `<nav>`), а
     не на весь `<header>`. Формально claim карточки/ARP ("клик вне меню закрывает") всё ещё
     верен буквально для кликов вне `<header>` целиком — поэтому не FAIL, но это реальный
     нюанс поведения, который стоит знать Architect/Product Owner на будущее (например, если
     позже в `<header>` добавится что-то ещё кликабельное рядом с меню).
5. Скриншоты `tester-light.png`/`tester-dark.png` (scratchpad) — вёрстка меню и Sidebar
   корректна в обоих режимах, контраст disabled-кнопок сохранён.

## Собственные ошибки в процессе проверки (прозрачность)

Как и предупреждает `tester.md`, риск случайной записи в реальную БД оказался не гипотетическим,
а реальным — в процессе отладки (до того, как была написана финальная версия скрипта с
собственной очисткой) несколько разрозненных диагностических запусков оставили 3 пустые заметки
на книге `"0. Wild Mind"` (id `"1"`). Обнаружено собственным `GET /api/workspace`, устранено
немедленно через `GET` + `PUT /api/workspace` с `ideas: []` для этой книги, проверено повторным
`GET`. Финальная версия скрипта дополнительно содержит собственный safety-net (поиск и очистка
любых заметок с текстом `"Проверка tester ..."`/`"DEBUG..."` в `finally`-блоке, независимо от
того, упал скрипт или нет) — при финальном прогоне сработал с результатом "нечего чистить".

## Постусловия — состояние реальной БД

Финальный `GET /api/workspace` сравнён с исходным снимком, снятым в самом начале этой проверки:

- Книга id `"1"` (`"0. Wild Mind"`, та единственная, что использовалась в CRUD-проверке) —
  **побайтово идентична** снимку "до".
- Единственное отличие во всей БД между снимком "до" и снимком "после" — `shortAnnotation`
  книги id `"3"` (`"2 Законы Тералии"`), которую этот тест не открывал и не трогал ни разу; текст
  вырос с одного слова до полноценной аннотации — это реальное редактирование в живом
  `npm run dev` Product Owner'а, произошедшее параллельно с этой проверкой, не последствие моих
  действий.
- Все временные серверы (`next start` порт 3418/3420/3421/3422, изолированный standalone-сервер
  порт 3423) остановлены, порты свободны. Временный скрипт-копия внутри `apps/studio/` удалён,
  `git status --short` не показывает следов.
