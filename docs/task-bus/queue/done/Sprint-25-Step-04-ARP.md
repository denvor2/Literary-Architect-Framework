## Что сделано

Для поля Title single generic-кнопка "AI" заменена на три типизированных quick-request кнопки —
"Подобрать аналоги" (comparables), "Мозговой штурм" (brainstorm), "Проверить на уникальность"
(uniqueness) — с сохранением единого response shape `{ suggestion, explanation }` для всех типов
запроса, как и требовала карточка. Развилка про UI для "проверить на уникальность" (нет кнопки
"Принять") реализована ровно так, как уже была решена в самой карточке — не переоткрывалась.

1. **`apps/studio/src/ai/operations.ts`** — новый экспортируемый тип
   `BookFieldRequestType = "comparables" | "brainstorm" | "uniqueness"`; `book_field_suggestion`'s
   payload получил опциональное поле `requestType?: BookFieldRequestType`.
2. **`apps/studio/src/ai/aiBus.ts`** — `requestType` прокинут в тело запроса к `/api/book-field`
   только когда присутствует (`...(requestType ? { requestType } : {})`), логика диспетчеризации
   по `operation.type` не тронута.
3. **`apps/studio/src/app/api/book-field/route.ts`** — добавлены `SUPPORTED_TITLE_REQUEST_TYPES`
   (валидация: неизвестное значение → 400) и `TITLE_REQUEST_PROMPTS` (по одному промпту на
   comparables/brainstorm/uniqueness). Выбор промпта: `requestType` учитывается только для
   `fieldName === "title"` — для остальных полей игнорируется (откат к их существующему generic
   промпту), что и защищает "не переписывать generic-кнопку для остальных полей" из Rules. Секция
   "Rules" в системном промпте тоже адаптирована — фраза "если текущее значение не пустое,
   улучши его" убрана для типизированных запросов (аналоги/брейншторм/уникальность — это не
   "улучшение текущего значения").
4. **`apps/studio/src/components/EditorArea.tsx`** — для Title:
   - убрана единственная кнопка "AI" и её завязка на пропсы
     `onRequestFieldSuggestion`/`fieldSuggestion` (эти пропсы остаются нетронутыми и по-прежнему
     используются для genre/premise/shortAnnotation/fullAnnotation — сравни: их JSX-блоки не
     тронуты вообще);
   - добавлены три pill-кнопки (`TITLE_QUICK_REQUESTS`, тот же визуальный паттерн, что
     `CRITIC_SUBCATEGORIES` в `AssistantPanel.tsx`) и **локальное** состояние компонента
     (`titleSuggestion`/`titleLoadingType`), которое вызывает `aiBus.execute()` **напрямую**
     (`import { execute as aiBusExecute } from "@/ai/aiBus"` — тот же модуль, что уже импортирует
     `page.tsx`), а не через lifted-to-`page.tsx` колбэки;
   - карточка suggestion для Title: кнопка "Принять" показывается для comparables/brainstorm
     (пишет `suggestion` в `Book.title` через существующий `onUpdateBook` проп — без изменений в
     этом пути) и **скрыта** для uniqueness — там только "Понятно" (закрыть), ровно как решено в
     самой карточке.
5. **`docs/adr/ADR-0011-book-field-operations.md`** — добавлена секция "Amendment (Sprint 25):
   типизированные quick-request кнопки для Title" (по образцу "Revision (Sprint 12)"/"Revision
   (Sprint 15 Step 01)" в ADR-0004) плюс короткая аннотация в шапке файла (`Status: Accepted,
   revised Sprint 25`) — Decision-текст оригинала не переписан.

### Технический выбор, не предусмотренный буквально в Allowed paths карточки

Карточка не включает `apps/studio/src/app/page.tsx` и
`apps/studio/src/workspace/useWorkspaceController.ts` в Allowed paths (второй явно в Forbidden).
Но именно `page.tsx` владеет существующим механизмом
`onRequestFieldSuggestion`/`fieldSuggestion` (там же — единственное место, откуда сегодня
вызывается `aiBus.execute()` для `book_field_suggestion`). Продеть `requestType` через этот
механизм означало бы обязательно менять `page.tsx` — что и запрещено картой, и рискованно
технически: `page.tsx` в этот момент параллельно редактируется другим агентом (Sprint-25-Step-02).
Решение (не эскалация — техническая реализация внутри уже согласованного направления): Title
получил собственное локальное состояние в `EditorArea.tsx`, вызывающее `aiBus.execute()` напрямую
— тот же самый модуль, тот же паттерн, что и `page.tsx` уже использует для генерик-кнопки других
полей, просто без похода через `page.tsx`. `page.tsx` не тронут ни на строку.

### Живая проверка (Shape 1) — важное замечание по безопасности

В процессе работы координатор передал срочное предупреждение: живая проверка Sprint-25-Step-01
случайно записала тестовые данные в реальную продовую Postgres-БД Product Owner (scratch-серверы
на любом порту используют тот же `DATABASE_URL`, что и обычный dev-сервер, согласно
dual-mode-хранилищу ADR-0012). Для этой карточки Rule Validation прямо требует клика "Принять"
по Title-предложению, который реально пишет в `Book.title` через `updateBook` — то есть ровно
операцию, которая могла задеть реальные данные.

Обнаружено дополнительно: в приложении **вообще нет функции удаления книги** (ни в
`useWorkspaceController.ts`, ни в `Sidebar.tsx`, ни в `DeveloperTools.tsx` — только
`deleteCharacter`/`deleteIdea`/`deleteThread`), то есть предложенный координатором план
"создать тестовую книгу через обычный UI → прогнать проверку → удалить её через собственное
действие удаления книги приложения" технически невыполним буквально — такого действия не
существует.

Решение (более сильная гарантия безопасности, чем создание+удаление реальной книги): scratch-
сервер запущен как **правильный** production-билд (`node .next/standalone/server.js` — не
`next start`, см. ниже) с **намеренно нерабочим** `DATABASE_URL`
(`postgresql://invalid:invalid@127.0.0.1:1/...`), переданным только как переменная окружения
этого процесса — `.env`/`.env.local` не редактировались. Из-за этого `/api/workspace` гарантированно
не может достучаться до реальной Postgres (проверено до, во время и после проверки — см. ниже), и
приложение по своей штатной dual-mode-логике (ADR-0012) работает исключительно через
`localStorage` — в дополнение к этому браузерный контекст Playwright свежий, в памяти, никогда не
сохраняется на диск. Итог: ни реальная БД, ни реальный профиль браузера Product Owner физически не
могли быть задеты — что сильнее, чем "создать и потом не забыть удалить одну тестовую книгу".

**Побочная находка при отладке:** первая попытка (`npx next start -p 3417`) реально предупредила
`"next start" does not work with "output: standalone" configuration` и после этого браузер получал
`404` на один JS-чанк и CSS — React не гидрировался, клики по кнопкам ничего не делали (симптом
не имел отношения к коду этой карточки). Исправлено запуском правильной команды для
`output: standalone` (Dockerfile, Sprint 22): `node .next/standalone/server.js` с `PORT=3417`,
`HOSTNAME=127.0.0.1`, плюс `ANTHROPIC_API_KEY`, переданным через переменную окружения процесса
(не редактируя `.env.local`). После этого гидратация и все клики заработали штатно.

**Предохранительные проверки:**
```
Preflight GET /api/workspace → {"ok":false,"error":"...Can't reach database server at 127.0.0.1:1"}
Финальная проверка GET /api/workspace (после всех кликов) → тот же ok:false, та же ошибка
```
— БД оставалась недоступной на всём протяжении проверки, ни один запрос не мог долететь до
реальных данных.

### Сценарий: реальный сервер, реальные ответы Claude (`verify-step04.mjs`, скрипт в scratchpad,
никогда не копировался в репозиторий)

Создана тестовая книга ("ТЕСТ Step-04 (не сохраняется в реальной БД)", премис про
принцессу-воительницу) исключительно в изолированном localStorage-режиме, все 3 quick-request
типа реально вызваны через настоящую кнопку в реальном UI:

```
PASS: Test book created and active (title input visible)
PASS: Comparables button visible
PASS: Brainstorm button visible
PASS: Uniqueness button visible
```

**Подобрать аналоги** — реальный ответ Claude, список аналогичных названий:
```
«Трон из костей и пепла», «Проклятая корона», «Изгнанница: Возвращение королевы», «Кровь и
корона», «Наследница проклятого трона»
```
explanation: "Эти названия отражают тему изгнанной принцессы-воительницы, борьбы за трон и
древнего проклятия, характерные для фэнтези о возвращении законной власти."
`PASS: карточка показывает "Принять"`.

**Мозговой штурм** — реальный ответ, один новый вариант названия:
```
Корона из пепла и терновника
```
`PASS: отличается от ответа "аналогов"`. Клик "Принять" → значение поля Title стало ровно
`"Корона из пепла и терновника"` — `PASS: Принять реально записал suggestion в Title через
updateBook`.

**Проверить на уникальность** — реальный аналитический вердикт (не вариант названия):
```
Похоже на существующие книги: заголовок использует распространённую в фэнтези формулу
«Корона/Трон из [материал] и [материал]» (например, ассоциации с «Троном из стекла» Сары Дж.
Маас и подобными young-adult фэнтези сериями), что снижает его отличимость.
```
`PASS: вердикт содержательно отличается от обоих предыдущих ответов`.
`PASS: карточка НЕ содержит кнопку "Принять"` (0 совпадений в DOM).
`PASS: карточка предлагает "Понятно" вместо неё` — клик закрывает карточку.

**Регрессия — generic-кнопка для Premise (не в scope этого шага, должна работать как раньше):**
```
PASS: генерик-кнопка "AI" у Premise по-прежнему присутствует
Premise suggestion: "Изгнанная принцесса-воительница возвращается в разорённое проклятием
королевство, чтобы отвоевать трон у узурпатора, прежде чем древняя тьма поглотит последних
жителей навсегда."
PASS: "Принять" по-прежнему пишет suggestion в поле Premise через updateBook (регрессии нет)
```

Итог скрипта: **`RESULT: 12 passed, 0 failed`**.

## Отклонения от Step Card

1. **Реализация Title-кнопок как локального состояния `EditorArea.tsx` вместо продевания через
   `page.tsx`-колбэки** (см. раздел выше) — технический выбор, вынужденный тем, что `page.tsx`
   вне Allowed paths этой карточки и параллельно редактируется другим агентом; сохраняет
   идентичное поведение для остальных полей и не меняет ни строки в `page.tsx`.
2. **Метод живой проверки отличается от буквального указания координатора** ("создать книгу
   через обычный UI → удалить через своё действие удаления книги") — по объективной причине:
   такого действия удаления книги в приложении не существует (обнаружено и задокументировано
   выше). Применена более сильная гарантия безопасности (заведомо недоступная реальная БД +
   память-only браузерный контекст) вместо создания и последующего удаления реальной записи.
3. **`npm run build`, повторно запущенный после живой проверки, не завершился зелёным** из-за
   транзиентной блокировки файловой системы (`EBUSY` на `.next/standalone`), не связанной с
   кодом этой карточки — см. раздел Validation. Опираюсь на первый успешный прогон build
   (сразу после кодовых правок, до ADR-правки, не влияющей на сборку).

Развилка про UI для "проверить на уникальность" — не отклонение, а прямое следование уже
принятому в самой карточке решению (нет кнопки "Принять", только "Понятно").
