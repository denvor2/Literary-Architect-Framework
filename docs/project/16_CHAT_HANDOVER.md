# 16. Chat Handover

## Назначение

Этот документ — инструкция для **новой сессии чата** (человека или AI), впервые
открывающей проект. Он не заменяет [15_MASTER_INDEX.md](15_MASTER_INDEX.md) и общий
`HANDOVER.md` — он говорит конкретно, что делать **в первые минуты новой сессии**,
прежде чем предлагать любое изменение.

## Порядок действий для новой сессии

1. Прочитать [15_MASTER_INDEX.md](15_MASTER_INDEX.md) — единственную точку входа.
2. Прочитать [11_CURRENT_STATE.md](11_CURRENT_STATE.md) — что реализовано сейчас, что нет,
   какой технический долг уже известен.
3. Прочитать [10_AI_BUS_WORKFLOW.md](10_AI_BUS_WORKFLOW.md) — правила ролей и формат ARP.
4. Проверить `docs/adr/` — какие архитектурные решения приняты (`Accepted`), какие только
   предложены (`Proposed`).
5. Только после этого — приступать к работе.

## Правила продолжения

- Новая сессия обязана считать [11_CURRENT_STATE.md](11_CURRENT_STATE.md) текущим
  состоянием проекта — не переоткрывать заново уже сделанную работу.
- Ничего не решать самостоятельно из разговора — если решение не зафиксировано в
  репозитории (ADR, PROJECT_STATE, CURRENT_SPRINT), оно не принято.
- Архитектура не меняется без явного одобрения Architect (ChatGPT).
- Каждый выполненный шаг завершается ARP-отчётом в формате из
  [10_AI_BUS_WORKFLOW.md](10_AI_BUS_WORKFLOW.md): STATUS, SUMMARY, FILES MODIFIED,
  VALIDATION, RISKS, SYSTEM STATE, NEXT STEP.

## Роли

- **Architect (ChatGPT)** — архитектура, приоритеты, ревью, приёмка.
- **Claude** — реализация, валидация (build/lint/runtime), отчётность. Не меняет
  архитектуру самостоятельно.
- **Product Owner** — приоритеты продукта, финальное утверждение scope.

## Если что-то не сходится

Если найдено противоречие между документами, между документом и кодом, или между
задачей и уже принятым решением — сессия обязана остановиться и сообщить об этом явно,
а не выбирать одну из версий по умолчанию.
