### 1. Модель пользователя (подтверждено Product Owner, 2026-07-10)

В приложении сегодня нет ни аутентификации, ни понятия "текущий пользователь" — ни в
useWorkspaceController.ts, ни в UI. При этом Prisma-схема (Sprint 23) требует непустой
Book.userId (обязательная связь с User).

**Решение для Sprint 24:** единственный локальный пользователь на инстанс, создаваемый
автоматически ("найти существующего или создать") при первом обращении к репозиторию, без
экрана логина — соответствует ADR-0003 "Phase 1 — single-user, local-first MVP". Это временное
состояние, не финальная модель — см. следующий пункт.

**Подтверждённое требование на будущее (не позже 5 спринтов от Sprint 24, ADR не должен
проектировать это сейчас — только зафиксировать срок как ограничение):** полноценная
мультипользовательская система с ролями Админ/Пользователь —
- Админ: все возможности пользователя, плюс создание/редактирование пользователей, блокировка
  новых регистраций, просмотр здоровья БД, снятие бэкапа (см. пункт 5 ниже).
- Пользователь: регистрация, авторизация, восстановление пароля, контрольные вопросы, капча.

Это НЕ входит в Scope этого ADR/Sprint 24 — зафиксировано в
docs/project/ROADMAP_18-27.md как новый Sprint (см. запись, добавленную 2026-07-10) именно
чтобы решение не потерялось. ADR-0012 должен сослаться на эту будущую точку явно (например, в
Consequences/Review Trigger), чтобы Sprint 24's User-модель была явно помечена как временная
заглушка, а не финальный дизайн.

### 2. Разделение персистентности между БД и localStorage (техническое, но зафиксировать явно)

Workspace (domain/workspace.ts) сегодня хранит вперемешку доменные данные (books) и
эфемерное UI-состояние (activeBookId, selectedChapterId, selectedSceneId,
selectedCharacterId, selectedAssistantMode) — оба сохраняются в один и тот же ключ
localStorage одним вызовом saveWorkspace(). Заголовочный комментарий prisma/schema.prisma
уже фиксирует намерение: "Workspace UI state ... is NOT persisted here — it is ephemeral". ADR
должен явно ратифицировать это как решение Sprint 24: в БД уходит только books; localStorage
продолжает хранить весь Workspace целиком, включая эфемерные поля, без изменений в этой части.

### 3. Форма API — грубая (whole-tree) или гранулярная по сущностям

Черновик дорожной карты буквально предполагает отдельные bookRepository.ts,
chapterRepository.ts и т.д., что по аналогии наводит на гранулярный REST (свой маршрут на
каждую операцию с каждой сущностью). Рекомендация: один "грубый" эндпоинт /api/workspace
(GET/PUT всего дерева books[] целиком), зеркалирующий уже существующий контракт
loadWorkspace()/saveWorkspace() — минимальная новая поверхность, соответствует принципу
"smallest validated step". Repository-слой при этом можно (и стоит) организовать по сущностям
внутри — это внутренняя декомпозиция, не меняющая внешний HTTP-контракт. Гранулярный REST можно
выделить позже, когда появится реальная причина (например, совместное редактирование или
частичные обновления) — не проектировать заранее.

### 4. Идентификаторы сущностей — блокирующая техническая находка (не вопрос выбора)

Chapter/Scene/Character/Idea/AssistantThread id в домене генерируются как String(nextNumber),
ЛОКАЛЬНО в пределах одной книги/главы (useWorkspaceController.ts): у первой главы ЛЮБОЙ книги
id "1"; у первой сцены ЛЮБОЙ главы id "1". Prisma-схема требует глобально уникального @id на
каждой из этих таблиц (не составного ключа с bookId). Без исправления первая же попытка
сохранить вторую книгу (или вторую главу) в БД упадёт на конфликте первичного ключа.
Зафиксировать это в ADR как контекст/предпосылку — само исправление вынесено в отдельный шаг
(Sprint-24-Step-02), выполняется до repository-слоя.

### 5. Dual-mode: обнаружение недоступности БД и поведение при рассинхроне (подтверждено
   Product Owner, 2026-07-10)

**Решение:**

- БД считается недоступной по результату КАЖДОЙ попытке чтения/записи (не только на старте
  сессии) — сессия может быть открыта часами, БД может упасть и восстановиться посреди неё.
- Неудачная запись в БД → тихий fallback на localStorage (без очереди повторов/бэкоффа в этом
  спринте — можно вынести в Sprint 25 "Production hardening" при реальной необходимости), НО
  **пользователь должен быть явно предупреждён** — видимое уведомление о рассинхроне/проблеме
  с БД (не тихо, как было в исходной рекомендации — Product Owner скорректировал именно этот
  пункт). Конкретная форма уведомления (toast/баннер/иконка статуса) — на усмотрение Step 06
  (UI-подключение), не фиксируется здесь как архитектурное решение.
- Рассинхрон между localStorage и БД разрешается "последняя запись побеждает" — при следующей
  успешной записи БД перезаписывается текущим books целиком (та же семантика, что уже
  используется для localStorage). Известное ограничение: конфликт из двух одновременных
  вкладок/устройств теряет более раннюю правку — приемлемо для single-user Phase 1.
- **Здоровье БД + возможность снять бэкап** — подтверждено как требование, но explicitly для
  Админ-роли, которая не существует до будущего мультипользовательского спринта (см. пункт 1).
  В Sprint 24 это НЕ реализуется (нет Admin UI, реализовывать не на чем) — зафиксировать в ADR
  как принятое требование к будущему Admin-функционалу, не как Step Card этого спринта.

### 6. Механизм переноса существующих данных

Буквальный scripts/migrate-localStorage-to-db.ts (Node CLI-скрипт) из черновика дорожной
карты нереализуем как отдельный скрипт — localStorage существует только в браузере, а
самостоятельной функции экспорта/импорта в проекте нет и она прямо отмечена как вне рамок
(docs/project/CURRENT_SPRINT.md, "Out of Scope... export/import формats"). Решение (не вопрос
выбора — единственный технически осуществимый вариант при нынешних ограничениях проекта): перенос
выполняется автоматически в браузере при загрузке приложения — если для текущего пользователя в
БД книг нет, а в localStorage они есть, клиент один раз выгружает их в БД через PUT
/api/workspace, дальше приложение работает в режиме "БД — основной источник".
