type: implementation

## Контекст

Product Owner попросил (устно, 2026-07-11) добавить форму поиска в верхнюю строку меню
(apps/studio/src/components/Header.tsx, добавлена Sprint-25-Step-01 как chrome-only заглушка —
Файл/Правка/Вид disabled, переключатель языка и "Войти" disabled) — поиск "по текстам книги
и книг в целом": (a) по содержимому текущей открытой книги, (b) по списку книг рабочего
пространства (найти/переключиться на книгу по названию). Product Owner явно попросил именно
одну форму ("форма для поиска", единственное число) — не два отдельных поля.

**Amendment (2026-07-11, после ответа Product Owner на единственный открытый вопрос этой
карточки).** Первая версия карточки оставляла Character/Idea вне поиска v1 и явно спрашивала,
включать ли их — не блокируя старт исполнения. Product Owner ответил: "Поиск должен быть
глобальным по всем элементам, но может имеет смысл добавить галочку — искать только в основном
тексте". Это меняет две вещи: (1) Character (имя/описание/заметки) и Idea (текст заметки)
активной книги теперь ТОЖЕ участвуют в поиске — весь текст карточки ниже переписан под это; (2)
добавляется переключатель "искать только в основном тексте" (сужает результаты до текста сцен).
Открытый вопрос закрыт, ниже он больше не фигурирует как открытый.

Это первый функциональный элемент в Header.tsx — сегодня компонент не принимает пропсов вообще
(export function Header()) и не имеет доступа ни к Workspace, ни к книгам, ни к персонажам/идеям.
Вся необходимая информация уже целиком в памяти клиента (useWorkspaceController.ts: books,
activeBookId, chapters = activeBook.chapters, characters, ideas — все уже возвращаются хуком и
уже используются в page.tsx) — новый API-роут/запрос к БД не нужен, поиск — чистый клиентский
substring-фильтр по уже загруженным данным.

Небольшая, полностью клиентская задача без изменений домена/схемы/бэкенда — поэтому один Step
Card, не последовательность "плумбинг -> UI".

## Разобранные технические развилки (решены самостоятельно, не эскалируются)

1. Форма UI: один input, общий выпадающий список с подписанными секциями ("Книги", "Главы и
   сцены", "Персонажи", "Идеи и заметки" — названия секций взяты из уже существующих заголовков
   этих же разделов в Sidebar.tsx/IdeasPanel.tsx для единообразия терминологии), а не несколько
   раздельных полей. Секция рендерится, только если в ней есть хотя бы один результат — пустые
   секции не показываются.
2. Логика поиска — чистая функция в apps/studio/src/domain/search.ts (новый файл), не размазана
   по Header.tsx. Тестируется Node-скриптом с фикстурными данными без браузера (Shape 2 из
   literary-studio-live-verify).
3. Что ищем (v1, после ответа Product Owner): Book.title — по всем книгам рабочего пространства
   (переключение книги, это единственная часть поиска, которая не ограничена активной книгой).
   У активной книги: Chapter.title, Chapter.subtitle, Scene.title, Scene.text, Character.name,
   Character.description, Character.notes (Character.photoUrl — не текстовое поле для поиска,
   исключено), Idea.text. Простое регистронезависимое вхождение подстроки (toLowerCase +
   includes) — никакой fuzzy-библиотеки и поискового индекса не добавляется.
4. Character/Idea ограничены активной книгой, а не всем рабочим пространством — как и
   Chapter/Scene (обе сущности вложены в конкретную Book в domain/model.ts, у них нет
   самостоятельного "какой книге принадлежит" контекста для показа в результатах, если бы
   искали по всем книгам сразу). Продолжение уже установленного в v1 деления: "Книги" — про
   переключение между книгами рабочего пространства; всё остальное — про содержимое ТЕКУЩЕЙ
   открытой книги. Product Owner просил "глобально по всем ЭЛЕМЕНТАМ" (то есть по типам данных
   — Character/Idea в дополнение к Chapter/Scene), не переформулировал изначальную рамку
   "текущая открытая книга" из исходной постановки задачи — если Product Owner имел в виду
   именно кросс-книжный поиск по персонажам/идеям, это отдельное расширение, не заявленное явно
   в его ответе.
5. Переключатель "искать только в основном тексте" — чекбокс в форме поиска, состояние чисто
   эфемерное (как query/isOpen), не персистится. Реализован как параметр чистой функции поиска
   (domain/search.ts), а не как отдельная UI-фильтрация после факта — при включённом
   переключателе функция поиска возвращает ТОЛЬКО совпадения в Scene.text активной книги (никаких
   Book.title, Chapter.title/subtitle, Character.*, Idea.* — это не "основной текст" книги, а
   метаданные/заметки).
6. **По умолчанию переключатель выключен** (полный глобальный поиск по всем элементам). Product
   Owner сформулировал "глобальным" как основное требование, а чекбокс — как "может имеет смысл
   добавить" для сужения, то есть дополнительную опцию поверх, а не наоборот. Выключенное
   состояние по умолчанию — то поведение, которое пользователь получит, ничего не трогая, и это
   должно быть максимально широким поиском согласно прямому запросу.
7. Переход к найденному месту переиспользует существующий механизм скролла Sprint 16-17
   (функция scrollBlockIntoView в Sidebar.tsx, id вида chapter-block-{id}/scene-block-{id}) для
   Chapter/Scene. Для Idea такого id сегодня нет (IdeasPanel.tsx рендерит список без атрибута
   id на обёртке каждой заметки, только React key) — требуется добавить один атрибут
   id={`idea-block-${idea.id}`} на обёртку каждой заметки в IdeasPanel.tsx, по той же самой
   конвенции именования, чтобы переиспользовать тот же generic scrollBlockIntoView без
   специального случая под Idea. Это единственное изменение в IdeasPanel.tsx — без переделки
   компонента.
8. Character не скроллится — переход к Character полностью переключает основной экран на
   CharacterPanel.tsx (уже существующий механизм: onSelectCharacter/selectCharacter, тот же,
   что использует Sidebar.tsx для списка персонажей) — это не блок внутри общего скролла,
   отдельный экран, поэтому доп. DOM-id и scrollIntoView для Character не нужны, CharacterPanel
   не редактируется.
9. Focus Mode прячет Sidebar.tsx целиком ({!isFocusMode && <Sidebar ...>} в page.tsx), а Idea
   рендерится только внутри Sidebar (перенесена туда Sprint-25-Step-01) — если Focus Mode
   активен в момент клика по результату-заметке, целевой DOM-узел physически не существует.
   handleSelectIdeaMatch в page.tsx обязан сначала выключить Focus Mode (setIsFocusMode(false)),
   если он включён, прежде чем скроллить — тот же принцип "сначала сделать цель видимой", что
   уже применяется к свёрнутым главам/сценам (см. п. 11).
10. Переключение книги переиспользует существующий selectBook/handleSelectBook — новый механизм
    не создаётся. Клик по активной книге в результатах ведёт себя как в сайдбаре (deselectAll).
11. Сворачивание/раскрытие перед скроллом (Chapter/Scene): если совпадение — внутри свёрнутой
    главы/сцены (collapsedChapterIds/collapsedSceneIds/isChaptersCollapsed, состояние в
    page.tsx), клик по результату обязан сначала развернуть нужный уровень, иначе scrollIntoView
    промахнётся или не найдёт узел в DOM вовсе.
12. Гонка рендера при переключении с CharacterPanel/другой книги на Chapter/Scene: целевой
    DOM-узел не существует в момент вызова обработчика (появится только после следующего
    рендера EditorArea) — scrollBlockIntoView вызывается не синхронно, а через
    requestAnimationFrame (двойной, если один недостаточно надёжен на практике — проверить
    живьём). Тот же принцип применяется к переходу на Idea после выхода из Focus Mode (п. 9) —
    Sidebar тоже должен успеть перерендериться, прежде чем идти scrollIntoView.
13. Минимальная длина запроса — 2 символа. Пустой/короткий запрос — список закрыт.
14. Сниппет текста (Scene.text/Character.description/Character.notes/Idea.text) в выпадающем
    списке — простая вырезка примерно 40 символов вокруг первого вхождения, без библиотек
    подсветки. Для Book.title/Chapter.title/Chapter.subtitle/Scene.title/Character.name —
    сниппет не нужен, само значение поля уже короткое, показывается целиком.

## Rules

- Простое регистронезависимое вхождение подстроки (toLowerCase + includes) — никакой
  fuzzy-библиотеки, никакого поискового индекса, никакого нового npm-пакета.
- Минимум 2 непробельных символа в запросе перед показом результатов.
- Секции выпадающего списка ("Книги"/"Главы и сцены"/"Персонажи"/"Идеи и заметки") — рендерятся
  только если в них есть результаты; названия секций совпадают с уже существующими заголовками
  тех же разделов в Sidebar.tsx/IdeasPanel.tsx.
- Чекбокс "искать только в основном тексте" — выключен по умолчанию (см. развилку 6); при
  включении domain/search.ts возвращает только совпадения Scene.text активной книги.
- Клик по результату закрывает выпадающий список после навигации.
- Escape и клик вне формы поиска закрывают выпадающий список — тот же паттерн, что уже есть в
  Header.tsx для Файл/Правка/Вид (mousedown-обработчик, menuBarRef).
- handleSelectSearchMatch (Chapter/Scene) обязан: снять "свернуть все главы"/свёрнутость нужной
  главы/сцены при необходимости, вызвать selectChapter/selectScene, скроллить через
  requestAnimationFrame (не синхронно).
- handleSelectIdeaMatch обязан: выключить Focus Mode, если он включён, затем скроллить через
  requestAnimationFrame (см. развилки 9, 12).
- Клик по результату-персонажу — прямой вызов уже существующего selectCharacter(characterId).
- Клик по результату-книге — прямой вызов уже существующего handleSelectBook(bookId).
- Клавиатурный шорткат фокуса поля поиска (Ctrl+K/`/`) — необязательное улучшение, включить,
  только если реализация тривиальна; пропустить и отметить в ARP, если не тривиально.
- UI-текст — по-русски, тем же визуальным языком, что остальной Header.tsx.
- Не переделывать структуру шапки — форма поиска добавляется как ещё один элемент строки.

  3. Свернуть главу с целевой сценой, повторить клик — глава разворачивается, скролл долистывает

  7. Включить чекбокс "искать только в основном тексте", повторить запрос, дающий совпадения и в
