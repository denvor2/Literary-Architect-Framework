# Словарь терминов Literary Studio

Краткий справочник по терминам и сокращениям, которые используются в переписке с Architect и
Programmer, а также в документации проекта (`docs/`). Составлен для программиста, впервые
подключающегося к процессу. Не заменяет [PROJECT_CHARTER.md](PROJECT_CHARTER.md) или
[HANDOVER.md](HANDOVER.md) — только поясняет термины, которые там встречаются.

## 1. Роли

| Термин | Расшифровка | Что это значит |
|---|---|---|
| **Product Owner** | — | Владелец продукта. Утверждает scope, приоритеты и финальное закрытие каждого Sprint-шага. |
| **Architect** | Chief Software Architect | Роль, отвечающая за архитектуру: ставит задачи (Step Card), принимает или отклоняет результат работы Programmer'а (REVIEW). Не пишет код сам. Роль, а не конкретная модель. |
| **Programmer** | Programmer (Executor); ранее — Lead Software Engineer | Роль, которая реализует задачи: пишет код, коммитит, готовит ARP. Роль, а не конкретная модель. |
| **Executor** | — | Синоним Programmer в документах AI Bus. |

## 2. Процесс разработки и AI Bus

| Термин | Расшифровка | Что это значит |
|---|---|---|
| **AI Bus** | — | Не часть продукта Literary Studio, а протокол работы между Architect и Programmer. Живёт в `docs/ai-bus/`. |
| **Sprint** | — | Крупный этап работы над продуктом. Состоит из нескольких Step'ов. Планируется и закрывается в `docs/project/CURRENT_SPRINT.md`. |
| **Step** | Sprint Step | Один шаг внутри Sprint — минимальная законченная единица работы. |
| **Step Card** | — | Файл-задание для Programmer'а: Objective, Scope, Validation, done_when. |
| **Scope** | — | Границы задачи: `allowed_paths` (что можно менять) и `forbidden_paths` (что нельзя). |
| **ARP** | Architecture/Task Review Package (отчёт о выполнении) | Отчёт Programmer'а после выполнения Step Card. |
| **REVIEW** | — | Ответ Architect'а на ARP: STATUS (OK/FIX/STOP), риски, следующий шаг. |
| **STATUS: OK** | — | Шаг сделан правильно, можно коммитить и двигаться дальше. |
| **STATUS: FIX** | — | Найдена конкретная проблема, тот же Step Card дорабатывается. |
| **STATUS: STOP** | — | Серьёзная проблема — работа останавливается до решения человека. |
| **Queue** | `docs/ai-bus/queue/` | Папка-очередь: `pending/` (ждёт), `active/` (в работе), `done/` (архив). |
| **Bootstrap** | — | Процедура «холодного старта» новой сессии — что и в каком порядке прочитать. |
| **Definition of Done** | DoD | Проверяемые условия завершённости шага/спринта. |
| **Техдолг** | технический долг | Осознанно отложенное упрощение в коде, не соответствующее целевой архитектуре. |

## 3. Документы проекта (`docs/`)

| Термин | Расшифровка | Что это значит |
|---|---|---|
| **ADR** | Architecture Decision Record | Документ одного архитектурного решения: контекст, решение, последствия, когда пересматривать. `docs/adr/`. |
| **Accepted** | статус ADR | Решение принято и действует. |
| **Proposed** | статус ADR | Предложено, но не проверено на практике. |
| **Superseded** | статус ADR | Заменено более новым ADR (старый не удаляется, помечается историческим). |
| **PROJECT_CHARTER.md** | — | Миссия, роли, принципы. Меняется редко. |
| **PROJECT_STATE.md** | — | «Живой снимок» состояния проекта, обновляется в конце каждого Sprint. |
| **CURRENT_SPRINT.md** | — | Что входит в текущий спринт и что уже сделано. |
| **CURRENT_STEP.md** | — | Указатель на активный Step Card. |
| **HANDOVER.md** | — | Быстрый старт для нового участника (человека или AI). |
| **DEVELOPMENT_WORKFLOW.md** | — | Весь процесс разработки: роли, как решения превращаются в код. |

## 4. Общие технические термины

| Термин | Расшифровка | Что это значит |
|---|---|---|
| **Репозиторий** | repository, «репо» | Хранилище кода и документации в git. |
| **Коммит** | commit | Сохранённый снимок изменений с сообщением. |
| **Push** | — | Отправка локальных коммитов на GitHub. |
| **Branch** | ветка (в проекте одна — `main`) | Отдельная линия разработки в git. |
| **`git status --short`** | — | Показывает изменённые/незакоммиченные файлы. |
| **Build** | сборка | Проверка, что код компилируется без ошибок (`npm run build`). |
| **Lint** | линтинг | Автопроверка стиля и типичных ошибок (`npm run lint`). |
| **MVP** | Minimum Viable Product | Минимальная рабочая версия продукта. |
| **UI** | User Interface | То, что видит и с чем взаимодействует пользователь. |
| **API / эндпоинт** | Application Programming Interface / endpoint | Точка обращения к серверу (например, `/api/line-editor`). |
| **Backend / Frontend** | — | Backend — сервер; Frontend — то, что работает в браузере. |

## 5. Термины продукта Literary Studio

| Термин | Расшифровка | Что это значит |
|---|---|---|
| **Expert** | AI Expert / Professional Role | Специализированный AI-помощник, выполняющий одну редакторскую роль. |
| **Expert Contract** | — | Формальное описание того, как Expert принимает запрос и отдаёт ответ. |
| **Domain Model** | — | Основные понятия: Book, Chapter, Scene, Workspace. |
| **Evolutionary Architecture** | — | Архитектурные решения выводятся из работающего кода, а не придумываются заранее. |
| **Line Editor** | — | Первый реализованный Expert — редактор грамматики и стиля. |
