## Sprint Goal

Bootstrap the first working piece of the Studio App using an evolutionary architecture
approach: get a real, running Next.js application in place before designing any contracts
(Expert system, prompts, workflows). Validate the toolchain and repository layout before
building on top of them.

## Completed Work

- Scaffolded the Studio App at `apps/studio/` using `create-next-app`:
  TypeScript, Tailwind CSS, App Router, ESLint, `src/` directory layout, `@/*` import alias.
- Verified the production build (`npm run build`) compiles cleanly with no errors.
- Verified the development server (`npm run dev`) starts and serves the default page —
  confirmed via HTTP 200 response and page title at `http://localhost:3000`.
- Reviewed the generated project structure with the Product Owner.
- Completed an Architecture Review pass and, as a result, produced the repository's first
  formal documentation set: [ADR-0001](../adr/ADR-0001-repository-structure.md),
  [ADR-0002](../adr/ADR-0002-expert-contract-vision.md), a rewritten
  [README.md](../../README.md), `docs/vision/` (roadmap, pricing, security, ideas), and
  `docs/project/` (charter, state, backlog, handover, workflow).

No AI integration, UI changes, Experts, workflows, prompts, authentication, or database work
were performed — these were explicitly out of scope for this sprint.

## Architectural Decisions

- **App location:** the Next.js Studio App lives in `apps/studio/`, not at the repository
  root, keeping the ecosystem (`framework/`, `prompts/`, `docs/`) and delivery layers separate
  and leaving room for further apps later. Recorded formally in
  [ADR-0001](../adr/ADR-0001-repository-structure.md).
- **No contracts designed yet.** Per the Product Owner's direction, this sprint deliberately
  avoided designing the Expert Contract before a working vertical slice exists. This is an
  explicit evolutionary-architecture choice: the first implementation (Sprint 04's Line Editor)
  will discover the contract; it does not implement one designed in advance. Recorded as vision
  in [ADR-0002](../adr/ADR-0002-expert-contract-vision.md), which remains in **Proposed**
  status until that discovery happens.

## Generated Files Review: `apps/studio/CLAUDE.md` and `apps/studio/AGENTS.md`

`create-next-app` now generates these automatically as part of its default template — this is
not something we requested or configured.

- **`apps/studio/AGENTS.md`** contains a short warning for AI coding agents: this Next.js
  version may differ from the agent's training data, and the agent should consult
  `node_modules/next/dist/docs/` before writing code. This is Next.js's own guardrail against
  AI tools hallucinating outdated APIs — it is specific to this app's framework version, not
  to the Literary Studio project.
- **`apps/studio/CLAUDE.md`** contains a single line: `@AGENTS.md` — a Claude Code file
  reference that simply re-exports the AGENTS.md content so Claude Code picks it up
  automatically when working inside `apps/studio/`.

**Does it duplicate the future repository-level CLAUDE.md?** No — they serve different
purposes. A root-level CLAUDE.md (still to be written) would describe the project's vision,
roles, phase roadmap, and the framework/app architectural boundary — repository-wide context.
This nested file is narrowly scoped to Next.js framework-version guardrails inside
`apps/studio/`. Claude Code loads CLAUDE.md files hierarchically (root plus nested), so both
can coexist and each would only add relevant, non-overlapping context. Recommend keeping both,
and letting the root CLAUDE.md link down to this one rather than repeating its content.

No files were deleted or modified as part of this review, per instruction.

## Sprint Outcome

Sprint 03 validated the project's foundations rather than shipping user-facing functionality:

- **Validated development environment** — Node/npm toolchain, `create-next-app` scaffold,
  production build, and dev server all confirmed working end-to-end.
- **Validated repository architecture** — the `apps/` boundary decided in Sprint 01/03 held up
  under real tooling constraints and is now formally recorded in ADR-0001.
- **Established engineering workflow** — this sprint introduced the Architecture Review step
  (Product Owner + Chief Software Architect reviewing documentation before commit), now
  captured in `docs/project/DEVELOPMENT_WORKFLOW.md`.
- **Created the repository's first architectural documentation** — two ADRs, a rewritten
  README, vision documents, and project-level documentation (charter, state, backlog,
  handover) now exist where previously there was only a three-line README.

## Lessons Learned

- Evolutionary architecture reduced premature abstraction — no Expert Contract was designed
  before there was anything real to learn from.
- Running an Architecture Review pass before committing documentation improved its quality and
  caught inconsistencies (e.g. outdated "no ADRs exist" statements) that a single-pass write
  would have missed.
- Repository documentation is becoming the project's source of truth — decisions made outside
  the repo (Sprint 02) are harder to act on until backfilled, which is direct evidence for
  keeping decisions in-repo going forward.
