## Контекст

Sprint 21 (ADR-0011, Accepted) добавил book_field_suggestion - одну generic AI-кнопку на поле
(title/genre/premise/shortAnnotation/fullAnnotation), возвращающую { suggestion, explanation }.
Реализация: apps/studio/src/app/api/book-field/route.ts (FIELD_PROMPTS - один промпт на поле),
apps/studio/src/ai/operations.ts (BookFieldName union + book_field_suggestion operation),
apps/studio/src/components/EditorArea.tsx (кнопка "AI" рядом с полем, карточка suggestion с
Принять/Отклонить).

Product Owner хочет, минимум для поля Title (возможно, для других полей позже - не в этом
шаге), несколько типизированных quick-request кнопок вместо одной generic: "подобрать аналоги",
"мозговой штурм", "проверить на уникальность" - названные PO как примеры, не исчерпывающий
список.

Per указанию из планирования этого спринта, это расширение существующей операционной
поверхности (доп. параметр + варианты промпта), не новая архитектура и не новый ADR - но ADR-
0011 должен получить короткую пометку об этом расширении (по образцу того, как ADR-0004
помечен "revised Sprint 12", ADR-0006 "revised Sprint 14").

## Развилка, которую можно решить самостоятельно (не эскалация - технический выбор внутри уже
## согласованного продуктового направления)

"Проверить на уникальность" по сути своей не предлагает значение поля для принятия - это
аналитический вердикт ("похоже на существующие книги X, Y" или "выглядит достаточно
уникальным"), а не альтернативный текст названия. Показывать его через существующую карточку
suggestion с кнопками "Принять"/"Отклонить" не имеет смысла продуктово - "принять" нечего
принимать в поле.

Решение (не эскалация, вытекает напрямую из уже принятого направления): сохранить единый
формат ответа { suggestion, explanation } для всех типов запроса (минимальное расширение), но
на UI-уровне не показывать кнопку "Принять" для аналитических типов запроса (uniqueness) -
только "Понятно"/закрыть карточку. Для "подобрать аналоги" и "мозговой штурм" - обычная
suggestion-карточка с Принять/Отклонить работает как есть (suggestion - предложенный текст,
который можно вставить в поле).

## Scope

Allowed paths:
- apps/studio/src/app/api/book-field/route.ts (доп. параметр запроса, варианты промпта per
  requestType)
- apps/studio/src/ai/operations.ts (book_field_suggestion payload gains optional requestType)
- apps/studio/src/ai/aiBus.ts (проброс нового поля, без изменения логики диспетчеризации)
- apps/studio/src/components/EditorArea.tsx (несколько quick-request кнопок вместо одной "AI"
  для Title; поведение suggestion-карточки для аналитических типов - см. выше)
- docs/adr/ADR-0011-book-field-operations.md (короткая пометка "revised Sprint 25" - см. ниже)

Forbidden paths:
- apps/studio/src/domain/**, apps/studio/src/storage/**,
  apps/studio/src/workspace/useWorkspaceController.ts
- Любые поля, кроме Title, в этом шаге - "возможно, другие поля позже" из формулировки Product
  Owner не значит "сделать сразу всё" - минимальный шаг, только Title.
- Не переписывать существующую generic-кнопку для полей genre/premise/shortAnnotation/
  fullAnnotation - они остаются как есть (одна generic AI-кнопка), не в scope этого шага.

## Rules

- Минимум 3 типа запроса для Title, соответствующих примерам Product Owner: "подобрать
  аналоги" (comparables), "мозговой штурм" (brainstorm), "проверить на уникальность"
  (uniqueness) - можно назвать поля/ключи request type иначе технически, но UI-подписи на
  русском должны соответствовать этим трём формулировкам или быть очевидно эквивалентными.
- requestType - опциональное поле; его отсутствие должно сохранять ровно текущее поведение
  (backward compatible, тот же принцип, что ADR-0011/ADR-0009 уже применяли).
- Ответ остаётся { suggestion, explanation } для всех типов - не вводить вариативный response
  shape ради uniqueness (см. решение выше).
- ADR-0011 должен получить отдельный раздел или пометку "Amendment (Sprint 25)" в конце файла -
  не переписывать существующий Decision текст задним числом, документировать расширение как
  добавление.
- UI copy - по-русски, тем же стилем, что остальные quick-action кнопки в проекте (например,
  CRITIC_SUBCATEGORIES pill-кнопки в AssistantPanel.tsx - используй тот же визуальный паттерн
  для консистентности, не изобретай новый).

## Validation

- npx tsc --noEmit, npx eslint src, npx prettier --check "src/**/*.{ts,tsx}", npm run build -
  чисто.
- Живая проверка (literary-studio-live-verify), реальный npm run dev + реальный вызов AI (не
  mock):
  1. Каждая из 3 quick-request кнопок для Title возвращает содержательно разный ответ
     (например, "подобрать аналоги" перечисляет похожие названия, "мозговой штурм" предлагает
     новый вариант, "проверить на уникальность" даёт вердикт) - убедиться разница видна в
     реальном ответе, не просто в том, что запрос отправился.
  2. Для "проверить на уникальность" карточка не показывает "Принять" (или показывает,
     дезактивированной/отсутствующей - как решено в реализации), только закрытие.
  3. Для "подобрать аналоги"/"мозговой штурм" Принять по-прежнему записывает suggestion в поле
     Title через updateBook (существующий путь, не должен регрессировать).
  4. Существующие поля (genre/premise/аннотации) продолжают работать через старую generic
     кнопку без изменений - regression-проверка.
- git status --short - только файлы из Allowed paths.
