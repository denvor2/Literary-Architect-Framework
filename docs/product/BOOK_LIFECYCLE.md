# Literary Studio — Book Lifecycle

## 1. Purpose

This document describes what a writer experiences as their book moves from a first idea to a
finished manuscript in Literary Studio — the stages they pass through, the help available at
each one, and the fact that they can always move freely between them. It describes none of the
mechanics behind that experience, only the experience itself.

## 2. Lifecycle Stages

### Idea

- **Goal:** capture the initial concept for a book before any real structure exists.
- **Typical user actions:** start a new book; note down the core idea or premise.
- **Typical assistants involved:** none required; a writer may optionally talk through the
  idea with Co-author.
- **Expected outcome:** a book exists with a basic premise captured.

### Planning

- **Goal:** shape the book's structure before writing begins in earnest.
- **Typical user actions:** fill in basic project information; sketch out chapters and the
  scenes within them.
- **Typical assistants involved:** Co-author, for help thinking through structure.
- **Expected outcome:** a chapter and scene outline the writer can start filling in.

### Drafting

- **Goal:** produce the actual text of the book.
- **Typical user actions:** write scenes directly, or work with an assistant to draft or
  continue a scene.
- **Typical assistants involved:** Co-author.
- **Expected outcome:** draft text exists for the book's chapters and scenes.

### Revision

- **Goal:** improve text that has already been drafted.
- **Typical user actions:** ask for edits on a passage; accept or reject the changes offered.
- **Typical assistants involved:** Editor.
- **Expected outcome:** polished text — still open to further change at any time.

### Review

- **Goal:** get an outside assessment of the manuscript's quality and impact.
- **Typical user actions:** ask for feedback on plot, characters, pacing, or overall
  impression.
- **Typical assistants involved:** Critic, Reader.
- **Expected outcome:** feedback the writer can choose to act on or set aside.

### Finalization

- **Goal:** bring the manuscript to a state the writer considers complete.
- **Typical user actions:** resolve remaining feedback; make final passes over the whole book.
- **Typical assistants involved:** Editor, Critic.
- **Expected outcome:** a manuscript the writer considers done.

### Publishing Ready

- **Goal:** confirm the manuscript is complete and polished.
- **Typical user actions:** one last full read-through and confirmation by the writer.
- **Typical assistants involved:** none required — this is the writer's own confirmation.
- **Expected outcome:** a finished manuscript. This stage is about the manuscript's
  completeness, not about the mechanics of publishing or distributing it (see Non-Goals).

## 3. User Flow

The normal happy path:

```
Create Book
   ↓
Fill basic information
   ↓
Plan
   ↓
Write chapters
   ↓
Iterate with assistants
   ↓
Review whole manuscript
   ↓
Finalize
   ↓
Export / Ready for publishing
```

## 4. Iteration Principle

The process is intentionally iterative, not a straight line. A writer may freely move between
stages, going back as often as they like. For example:

```
Draft
  ↓
Critic
  ↓
Editor
  ↓
Draft
  ↓
Reader
  ↓
Draft
  ↓
Final
```

There is no rule that a book must pass through the lifecycle stages exactly once, or in order.
Moving backward — from Review back to Drafting, for instance — is a normal, expected part of
writing a book, not an exception.

## 5. Non-Goals

Explicitly outside MVP:

- Illustrations
- Merchandise
- Publishing integrations
- Marketing
- Payments

## 6. Design Principles

- Books are living projects.
- Writing is iterative.
- Assistants support the writer.
- The writer always makes the final decisions.
- No assistant owns the manuscript.
