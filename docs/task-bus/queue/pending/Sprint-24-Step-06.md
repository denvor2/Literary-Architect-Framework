## Контекст

Финальный шаг Sprint 24 — подключает асинхронный workspaceStorage.ts (Sprint-24-Step-05) к
единственному месту, которое его вызывает. Definition of Done Sprint 24
(docs/project/ROADMAP_18-27.md) не включает новых визуальных элементов — этот шаг чисто
технический, поведение приложения для пользователя не меняется.

## Objective

Адаптировать восстанавливающий/сохраняющий useEffect в хуке под асинхронные
loadWorkspace()/saveWorkspace() (Step 05), не меняя внешнее поведение приложения:

- Восстановление при монтировании: loadWorkspace() теперь возвращает Promise<Workspace> —
  вызвать её внутри существующего useEffect, дождаться результата, затем setWorkspace(...)/
  setIsLoaded(true), как сегодня. Между монтированием и разрешением промиса UI продолжает
  показывать EMPTY_WORKSPACE (как сегодня до окончания синхронного loadWorkspace()) — не
  добавлять отдельный "loading"-экран, это не входит в scope.
- Сохранение при каждом изменении workspace: тот же useEffect, что и сегодня, но
  saveWorkspace(workspace) теперь возвращает Promise — вызвать её, не блокируя рендер
  (.catch(() => {}) или эквивалент — защита от unhandled rejection на вызывающей стороне;
  сама функция уже гарантированно не бросает наружу по контракту Step 05). Debounce/
  троттлинг сетевых PUT-запросов НЕ добавлять в этом шаге — каждое изменение workspace
  по-прежнему инициирует попытку записи, как сегодня для localStorage. Заметность сетевой
  задержки при активном наборе текста в сцене — реальный возможный вопрос производительности,
  но технического характера; если живая проверка ниже покажет, что это заметно на практике —
  зафиксировать в ARP как известный открытый пункт для отдельного узкого шага позже (например,
  Sprint 25), не решать спекулятивно здесь.

## Validation

- npx tsc --noEmit, npm run lint, npx prettier --check, npm run build — чисто (это последний
  шаг цепочки, build должен полностью пройти без ожидаемых ошибок).
- npm run test:e2e (Playwright, apps/studio/e2e/smoke.spec.ts) — все 12 тестов зелёные, включая
  сценарий localStorage persistence — первая живая проверка, что переход на dual-mode не
  сломал ни один существующий сценарий.
- Живая проверка (реальный сервер + реальная БД, docker compose up postgres, npm run dev):
  1. Открыть приложение, создать книгу/главу/сцену, ввести текст в сцену -> перезагрузить
     страницу -> данные восстановились.
  2. Остановить контейнер postgres, продолжить редактировать -> приложение не падает, данные
     сохраняются в localStorage (проверить через DevTools).
  3. Запустить postgres обратно, перезагрузить страницу -> данные из БД (последнее успешное
     состояние до остановки) отображаются; если между остановкой и перезапуском БД
     вносились правки при недоступной БД — явно зафиксировать в ARP их фактическую судьбу
     (по ADR-0012 они должны восстановиться из localStorage при следующей загрузке до первого
     успешного обращения к БД, а затем перезаписать БД целиком при следующем сохранении —
     описать фактически наблюдаемое поведение, не ожидаемое).
- Приложить в ARP реальные логи/скриншоты каждого сценария.
