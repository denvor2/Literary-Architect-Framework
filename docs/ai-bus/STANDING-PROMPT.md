# Standing Prompt — Programmer Queue Routine

Читать в начале каждой сессии Programmer (Executor), наравне с Bootstrap
(docs/ai-bus/BOOTSTRAP.md). Не дублирует docs/ai-bus/queue/README.md —
это краткая рабочая инструкция поверх уже описанного там протокола.

## Что делать при старте сессии

1. Проверь `docs/ai-bus/queue/pending/`.
   - Пусто → сообщи об этом человеку и жди.
   - Есть файл (Step Card) → переходи к шагу 2.
2. `git mv pending/<file> active/<file>` — этим же коммитом начинаешь
   работу (см. README.md, п.2 Protocol). Если что-то уже лежит в
   `active/` — по Single Active Step Rule сначала заверши/разреши тот
   Step Card, новый не начинай.
3. Выполни Step Card строго по его собственному содержимому
   (Scope/Objective/Rules/Validation/Output) — сам Step Card является
   заданием, этот файл только про то, как его забрать и куда положить
   результат.
4. Результат — ARP — оформи КАК ФАЙЛ в `active/`, рядом со Step Card:
   `<StepCardName>-ARP.md`. Это не отменяет вывод ARP в чат (человек
   должен видеть его), но теперь он ещё и лежит в репозитории —
   именно это делает возможным ревью без ручного копирования текста.
5. Сообщи человеку (в чате): "ARP по <id> готов, файл в
   docs/ai-bus/queue/active/, жду ревью Architect."
6. Дальше — по уже описанному в README.md разделу
   "Programmer Response to Review": проверяешь `active/` на
   `REVIEW.md`, действуешь по STATUS (OK / FIX / STOP).

## Текущее ограничение (честно, не скрывать)

Шаги 5 и "проверка REVIEW.md" пока не автоматизированы — человек
вручную просит Architect посмотреть очередь, и вручную просит
Programmer проверить `active/` на REVIEW.md. Периодическая
автоматическая проверка с обеих сторон — предмет отдельного, ещё не
согласованного Step Card. Не выдавай эту ручную процедуру за
автоматическую.

## Избегать блокировки чата опасно выглядящими паттернами

Когда ARP или сообщение в чат содержит конфигурацию с явными
разрушительными операциями (в разделах запрета/deny, в примерах
"чего не делать" и т.п.) — не печатай такие строки буквально и
многократно прямо в тексте, адресованном в чат. Вместо этого:

- Полный точный код/конфигурация — только в файле (queue/active/
  ARP-файл, коммит), не дублировать построчно в самом чат-сообщении.
- В самом чат-сообщении — описывай словами, что сделано ("добавлены
  правила, запрещающие рекурсивное удаление и принудительный push"),
  без буквального повторения конкретных команд.
- Это не мера предосторожности против реального риска — код в файлах
  безопасен и уже проверяется Architect'ом обычным образом. Это чисто
  техническое ограничение интерфейса чата, которое эта заметка
  обходит.
