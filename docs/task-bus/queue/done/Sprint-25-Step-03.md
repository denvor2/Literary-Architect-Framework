## Контекст (сокращённо, полная версия - в ADR-0013)

Product Owner попросил (2026-07-11) добавить каждому из 4 режимов помощника (Соавтор/Редактор/
Критик/Читатель) "gear"-иконку, открывающую диалог настройки: отображаемое имя, промпт, и набор
"типовых запросов". ADR-0013 фиксирует целевую архитектуру: Admin задаёт дефолт для каждого
режима (имя/промпт/типовые запросы), обычный пользователь может настроить свою версию ТОЛЬКО
если ему разрешено (право или тариф - механизм не уточнён, не блокирует этот шаг).

## Обязательное ограничение реализации этого шага - см. ADR-0013, раздел "Implementation
## constraint for Sprint 25 Step 03" - процитировано здесь дословно, не переформулировать

Реального различия Admin/User в проекте ещё нет (ADR-0012 - единственный локальный пользователь-
заглушка на весь Phase 1 MVP; настоящие роли приезжают только со Sprint 29/ADR-0015, ещё не
написан). **Реализация этого шага обязана трактовать текущего единственного пользователя как
имеющего полный (Admin-эквивалентный) доступ** - имя/промпт/типовые запросы КАЖДОГО режима
напрямую редактируемы этим единственным пользователем, БЕЗ какого-либо UI проверки прав -
проверять нечего, второй роли не существует. **Не строить никакую заглушечную систему прав/
тарифов заранее** - это было бы спекулятивным проектированием возможности, для которой в
проекте ещё нет ни одного рабочего примера (evolutionary architecture, ADR-0002). Когда Sprint 29
введёт настоящих пользователей, именно он обязан смигрировать это состояние "всё напрямую
редактируемо" в "дефолт Admin + gated override" - это явно помечено как забота Sprint 29, не
решается здесь.

## Хранилище (по ADR-0013, раздел Storage scope)

Одна запись на AssistantMode (coauthor/editor/critic/reader), общая на весь инстанс - НЕ per
Book, НЕ per отдельный (пока не существующий) User. Практически для этого шага: одна новая
Prisma-модель (или расширение существующей, на усмотрение реализации) с полями примерно
`{ assistantMode, displayName, promptSuffix, typicalRequests[] }`, без привязки к Book/User -
4 записи максимум (по одной на режим), редактируемые напрямую единственным пользователем.

## Override-семантика (по ADR-0013, раздел Override semantics - append, not replace)

Кастомный промпт (`promptSuffix`) ДОБАВЛЯЕТСЯ к существующему hardcoded системному промпту
каждого route.ts (тот же паттерн, что уже применяют CRITIC_SUBCATEGORY_PROMPTS поверх
CRITIC_BASE_PROMPT, ADR-0009) - НЕ заменяет его целиком. Это защищает существующие контракты
формата ответа (Critic возвращает structured reviews[], Co-author structure - JSON) от поломки
свободным текстом пользовательского промпта. Отображаемое имя - чистый UI-лейбл, не уходит в
модель вообще. Типовые запросы - preset-кнопки, заполняющие поле ввода чата (тот же UX-паттерн,
что уже есть у CRITIC_SUBCATEGORIES pill-кнопок в AssistantPanel.tsx) - не новая AI Bus операция.

## Scope

Allowed paths:
- apps/studio/prisma/schema.prisma (новая модель для assistant-настроек, миграция)
- apps/studio/src/repositories/** (CRUD для новой модели, по образцу существующих
  userRepository.ts/bookRepository.ts)
- apps/studio/src/app/api/** (новый маленький эндпоинт для чтения/записи assistant-настроек;
  ИЛИ расширение существующих 4 Expert route.ts, чтобы они читали promptSuffix и добавляли его
  как suffix к своему system-промпту - обе части нужны)
- apps/studio/src/components/AssistantPanel.tsx (gear-иконка на каждом режиме, диалог
  настройки: имя/промпт/типовые запросы)
- apps/studio/src/ai/operations.ts, apps/studio/src/ai/aiBus.ts (только если требуется новая
  операция для чтения/записи настроек - предпочтительно обычный REST эндпоинт, не AI Bus
  операция, так как это не AI-вызов)

Forbidden paths:
- apps/studio/src/domain/** (Book/Chapter/Scene/Character/Idea/Workspace типы не меняются -
  assistant-настройки не часть Book, см. Storage scope выше)
- Любая система прав/ролей/тарифов - вне scope, см. ограничение выше.
- Rendering-ветки конкретных режимов (сам чат Co-author/Editor/Critic/Reader) - меняется только
  добавление gear-иконки и её диалога, не логика отправки сообщений.

## Validation

- npx tsc --noEmit, npx eslint src, npx prettier --check "src/**/*.{ts,tsx}", npm run build -
  чисто.
- Живая проверка (literary-studio-live-verify), реальная БД + реальный AI-вызов (не mock):
  1. Открыть gear-диалог для одного режима (например, Critic), задать кастомное имя, промпт-
     suffix и хотя бы один типовой запрос, сохранить.
  2. Убедиться, что кастомное имя отображается в UI вместо стандартного (MODE_META.label).
  3. Отправить реальный запрос через этот режим - убедиться, что ответ AI отражает влияние
     кастомного promptSuffix (например, попросить в промпте что-то заметно отличающее поведение,
     проверить что это видно в реальном ответе Claude, не просто что запрос ушёл).
  4. Убедиться, что структура ответа (JSON reviews[] для Critic и т.п.) не сломалась несмотря на
     кастомный suffix - append, не replace, работает как задумано.
  5. Режим без кастомизации (например, Reader, если не трогали) продолжает работать как раньше -
     regression-проверка.
- git status --short - только файлы из Allowed paths.
