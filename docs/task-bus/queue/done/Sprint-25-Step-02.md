## Часть 1 - перетаскиваемый делитель (решение принято)

Product Owner подтвердил: главная рабочая область делится примерно пополам - текст/редактор
слева, чат помощника справа (это в целом уже так, крупными мазками, через текущую раскладку
EditorArea/AssistantPanel), с делителем, который можно перетаскивать мышью, по умолчанию 50/50.

### Развилка зависимости - решить самостоятельно по факту, не спрашивать

В проекте нет ни одной существующей resizable-панели библиотеки. Два варианта:
(а) добавить одну (react-resizable-panels - самая распространённая, активно поддерживаемая,
небольшой размер, соответствует ADR-0003 "официальные SDK/минимум зависимостей" в духе, а не
букве - это не AI-оркестрация, а обычная UI-библиотека, не подпадает под ограничение ADR-0003
про AI-фреймворки);
(б) реализовать вручную через mousedown/mousemove/mouseup на разделительной полосе (без новой
зависимости, но больше кода на поддержку - drag state, min/max ширина, touch-события).

Рекомендация: (а) - react-resizable-panels, потому что делитель с перетаскиванием - это ровно
тот случай, где "не изобретать велосипед" оправдан (учёт touch-устройств, доступности,
edge-кейсов при resize окна), а сама библиотека маленькая и не создаёт архитектурного риска
(чистый UI-примитив, не AI-зависимость). Это не "молчаливое добавление зависимости" в смысле
Правил проекта - зафиксировать выбор явно в ARP этого шага и почему, чтобы Product Owner видел
решение постфактум, даже если сам выбор технический, не продуктовый.

Персистентность позиции делителя: НЕ входит в это решение как обязательное - если реализация
добавит её тривиально (например, через localStorage, тем же паттерном, что уже используют
collapsedChapterIds и подобные эфемерные UI-состояния), это нормально; если нет - оставить
делитель сбрасывающимся на 50/50 при перезагрузке, не усложнять ради этого.

## Часть 2 - реорганизация picker-а 4 режимов (решение принято, не эскалация)

Сейчас AssistantPanel.tsx рендерит 4 режима (Соавтор/Редактор/Критик/Читатель) как
grid-cols-2 сетку карточек (описание + emoji, см. MODE_META/ASSISTANT_MODES, строки ~729-754) -
занимает существенную вертикальную площадь панели шириной ~320px.

**Решение Product Owner:** заменить эту сетку карточек на ряд/сетку КВАДРАТНЫХ иконок-кнопок
(компактные, одинаковой ширины и высоты, без видимого текста внутри самой кнопки), где подпись
режима (текущий MODE_META.label, например "Соавтор") показывается только при наведении мыши
(hover tooltip - нативный `title`-атрибут или лёгкий кастомный tooltip, на усмотрение
реализации, без новой зависимости ради одного tooltip). Активный режим визуально выделяется
(тот же принцип выделения, что сейчас через `info.activeBorder`/`isActive`, просто в новой
квадратной форме, не в карточке).

Текстовое описание режима (`meta.description`, сейчас показывается в карточке picker-а)
переносится под ряд иконок, показывается только для ТЕКУЩЕГО активного режима (не для всех 4
сразу, как сейчас) - экономит вертикальное место, которое и было целью Product Owner.

### Координация с содержимым иконки (важно, не изобретать заново)

Сама square-кнопка на этом шаге ПРОДОЛЖАЕТ использовать текущий emoji-глиф из
`MODE_META.emoji` (🟡🟢🔴🔵) как содержимое - замена emoji на настоящий icon-set (например,
lucide-react) - отдельная забота Sprint-25-Step-05 (design pass, ui-specialist), не этого шага.
Разделение работы: этот шаг меняет СТРУКТУРУ (квадратные кнопки + hover tooltip вместо сетки
карточек с описанием), Step 05 позже меняет СОДЕРЖИМОЕ иконки (emoji -> настоящий icon-компонент)
внутри уже готовой структуры - не переделывать структуру дважды. Обе карточки описывают один и
тот же визуальный язык (квадратные icon-кнопки + hover-подпись) - см. также обновление в
Sprint-25-Step-05.

### Форма разметки (техническая деталь, решить самостоятельно)

Ряд из 4 одинаковых квадратных кнопок (например, `w-10 h-10` или похожий фиксированный
квадратный размер, на усмотрение реализации) в одну строку (не 2 колонки, как сейчас) - при
узкой ширине панели (320px, `lg:w-80`) 4 кнопки в ряд помещаются свободно, uместнее одной
строки, чем прежней 2-колоночной сетки. `ASSISTANT_MODES.map(...)` - структура перебора
остаётся, меняется только JSX разметка каждой кнопки и её контейнера.

## Rules

- Делитель по умолчанию 50/50; перетаскивание мышью в разумных пределах (не давать сжать
  панель до нуля - минимальная ширина с каждой стороны, на усмотрение реализации).
- Если добавляется react-resizable-panels (или любая другая resizable-библиотека) - явно
  зафиксировать в ARP какая и почему, не добавлять её "тихо" в package.json без упоминания.
- Picker: квадратные кнопки-иконки в ряд, hover tooltip с подписью режима, описание активного
  режима под рядом - см. Часть 2 выше. Не использовать вкладки, не оставлять текущую 2-
  колоночную сетку карточек.
- Не менять содержимое иконки (emoji остаётся до Step 05) - только структуру/форму кнопки.
- UI copy - по-русски, тем же стилем, что остальной проект.

## Validation

- npx tsc --noEmit, npx eslint src, npx prettier --check "src/**/*.{ts,tsx}", npm run build -
  чисто.
- Живая проверка (literary-studio-live-verify):
  1. Делитель: реальный npm run dev, перетащить границу мышью (или через Playwright
     drag-эмуляцию), убедиться что ширина обеих панелей меняется, скриншот до/после.
  2. Picker: скриншот новой раскладки (ряд квадратных кнопок), наведение мышью показывает
     tooltip с названием режима (например, через Playwright hover + проверка видимости
     tooltip/title), переключение между режимами по-прежнему работает (существующее поведение
     onSelectMode не регрессирует), описание видно только для активного режима.
- git status --short - только файлы из Allowed paths.
