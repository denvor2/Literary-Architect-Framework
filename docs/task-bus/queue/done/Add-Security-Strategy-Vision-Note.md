## Objective

Добавить новый раздел дословно:

```markdown
## 13. Стратегия безопасности (обсуждено, не реализовано)

Обсуждено с Product Owner 2026-07-06. Ключевой принцип: защита
должна соответствовать реальной поверхности атаки на данном этапе,
а не строиться заранее для гипотетической архитектуры. Сейчас в
проекте нет ни базы данных, ни аккаунтов, ни бэкенд-состояния — вся
поверхность атаки, актуальная для полноценного веб-продукта, ещё
не существует. Полный список требований — раздел 10.

**Важная данность, которую нужно осознавать:** текст книги уже
сейчас уходит на сервер Anthropic открытым текстом при каждом
вызове Line Editor/Critic/Reader — это неотъемлемая часть работы
AI-функций, не недоработка. "Сервер не может прочитать книгу"
недостижимо без отказа от AI-редактуры как таковой.

**Что делать сейчас, независимо от этапа:**
- Не доверять входным данным на API-роутах (уже практикуется)

**Что добавить непосредственно перед публичным запуском (дополнение
к разделу 10):**
- Rate limiting на API-роуты — без этого публичный доступ без
  аутентификации может использоваться для накрутки счёта за
  Anthropic API кем угодно, это не про кражу данных, а про деньги.

**Стратегия на случай кражи БД (когда БД появится):**
- Шифрование данных "at rest" (стандартная практика) — защищает от
  кражи файла/дампа базы (утечка бэкапа, неверно настроенный доступ
  к облаку). Разумный стандарт на момент появления БД.
- Полная "невидимость данных для сервера" (шифрование ключом,
  привязанным к паролю пользователя) — сознательно НЕ выбрана как
  цель: плохо совместима с AI-функциями (сервер должен видеть текст,
  чтобы дать AI-отзыв) и усложняет восстановление доступа (забытый
  пароль = потерянная книга без возможности сброса).

**Статус:** согласованный принцип, не реализовано — относится к
разделу 10 "Требования перед запуском в интернет", не к текущей
разработке.
```
